Browser Context — The Foundation

Think of Playwright's 3-layer architecture like a bank building: Browser = the building, BrowserContext = isolated private cabin, Page = the desk where work happens.

🏗️ Browser

The browser process. Just infrastructure — no tabs, no sessions.

typescript
const browser = await chromium.launch();
🚪 BrowserContext

Isolated session — own cookies, localStorage, auth. Like incognito window.

typescript
const context = await browser.newContext();
📄 Page

The actual tab. Navigate, click, type, assert — all through page.

typescript
const page = await context.newPage();
typescript — Full Context Flow
// Multi-user isolation — testing Admin + User simultaneously
const contextA = await browser.newContext();
const pageA    = await contextA.newPage();  // Admin session

const contextB = await browser.newContext();
const pageB    = await contextB.newPage();  // Regular User — fully isolated

// Context options
const context = await browser.newContext({
  baseURL:           'https://myapp.com',
  viewport:          { width: 1280, height: 720 },
  locale:            'en-IN',
  timezoneId:        'Asia/Kolkata',
  storageState:      'auth.json',      // reuse login session!
  ignoreHTTPSErrors: true,
});

JS Popups & Window Switching

⚠️
Always register the dialog listener BEFORE the action that triggers it. Register after = too late, dialog already fired.
typescript — All 3 Dialog Types
// Alert — just OK
page.on('dialog', async dialog => {
  console.log(dialog.type());    // 'alert' | 'confirm' | 'prompt'
  console.log(dialog.message());
  await dialog.accept();           // click OK
});

// Confirm — OK or Cancel
page.on('dialog', async dialog => {
  await dialog.accept();    // OK  → returns true
  // await dialog.dismiss(); // Cancel → returns false
});

// Prompt — type + OK
page.on('dialog', async dialog => {
  await dialog.accept('Sumit Sagar');  // type text + OK
});

// New Tab / Window — Promise.all is critical!
const [newPage] = await Promise.all([
  context.waitForEvent('page'),            // listen first
  page.click('a[target="_blank"]')           // then click
]);
await newPage.waitForLoadState();
await page.bringToFront();             // switch back

Assertions Deep Dive

The golden rule: DOM/browser involved → await expect(). Plain JS value → expect() no await.

06

Assertion Categories

🌐 Async — DOM-based (await)
typescript
// Element STATE
await expect(el).toBeVisible();
await expect(el).toBeHidden();
await expect(el).toBeEnabled();
await expect(el).toBeDisabled();
await expect(el).toBeChecked();
await expect(el).toBeFocused();
await expect(el).toBeEmpty();

// Element CONTENT
await expect(el).toHaveText('exact');
await expect(el).toContainText('partial');
await expect(el).toHaveValue('input val');
await expect(el).toHaveCount(5);

// Element ATTRIBUTES
await expect(el).toHaveAttribute('type', 'email');
await expect(el).toHaveClass('btn-primary');
await expect(el).toHaveCSS('color', 'rgb(0,0,0)');

// PAGE level
await expect(page).toHaveURL('/dashboard');
await expect(page).toHaveTitle('App');
📦 Sync — JS Values (no await)
typescript
// Numbers
expect(count).toBe(5);
expect(count).toBeGreaterThan(0);
expect(count).toBeLessThan(10);
expect(count).toBeGreaterThanOrEqual(3);

// Strings
expect(text).toBe('Welcome');
expect(text).toContain('Welcome');
expect(text).toMatch(/welcome/i);

// Booleans
expect(isVisible).toBeTruthy();
expect(isDisabled).toBeFalsy();

// Arrays
expect(arr).toHaveLength(3);
expect(arr).toContain('b');
expect(arr).toEqual(['a', 'b', 'c']);

// Negation
expect(count).not.toBe(0);
expect(text).not.toContain('Error');
🪶 Soft Assertions — Don't Stop on Failure
typescript
// Soft = continue even if these fail
await expect.soft(page.getByText('Welcome')).toBeVisible();
await expect.soft(page.locator('.badge')).toContainText('3');

// Hard assertion at end — stops if any soft assertion failed
expect(test.info().errors).toHaveLength(0);
🧠 Memory Tricks
toBe → IS IT? (state/identity) toHave → DOES IT OWN? (property/content) toContain → INSIDE? (partial match) await → BROWSER? (DOM involved = auto-retry) no await → VALUE? (plain JS = instant check)

typescript — fixtures/auth.fixture.ts
import { test as base } from '@playwright/test';

type AuthFixtures = {
  loggedInPage: Page;
  adminPage:    Page;
};

export const test = base.extend<AuthFixtures>({

  loggedInPage: async ({ page }, use) => {
    // SETUP — runs before test
    await page.goto('/login');
    await page.getByLabel('Email').fill('user@test.com');
    await page.getByLabel('Password').fill('Pass@1234');
    await page.getByRole('button', { name: 'Login' }).click();
    await page.waitForURL('/dashboard');

    await use(page);  // ← test runs here (green light)

    // TEARDOWN — runs after test
    await page.goto('/logout');
  }
});

DB Validation — PostgreSQL

typescript — DbHelper + Full Stack Test
// utils/DbHelper.ts
import { Pool } from 'pg';

export class DbHelper {
  private pool: Pool;

  constructor() {
    this.pool = new Pool({
      host:     process.env.DB_HOST,
      database: process.env.DB_NAME,
      user:     process.env.DB_USER,
      password: process.env.DB_PASSWORD,
    });
  }

  async queryOne(sql: string, params: any[] = []) {
    const client = await this.pool.connect();
    try {
      const result = await client.query(sql, params);
      return result.rows[0] || null;
    } finally { client.release(); }
  }
}

// Full stack test — UI + API + DB
test('register user — verify in UI, API and DB', async ({ page, request, db }) => {
  // 1. UI action
  await page.goto('/register');
  await page.getByLabel('Email').fill('sumit@test.com');
  await page.getByRole('button', { name: 'Register' }).click();

  // 2. API verify
  const apiRes = await request.get('/api/users?email=sumit@test.com');
  expect(apiRes.status()).toBe(200);

  // 3. DB verify — ground truth
  const dbUser = await db.queryOne(
    `SELECT * FROM users WHERE email = $1`,
    ['sumit@test.com']
  );
  expect(dbUser).not.toBeNull();
  expect(dbUser.is_active).toBe(true);
});

.env & dotenv — Explained

.env = the FILE (stores secrets as key=value text). dotenv = the LIBRARY (reads .env and loads into process.env). They work together — dotenv is useless without .env, and .env values don't reach your code without dotenv.

typescript — dotenv patterns
import dotenv from 'dotenv';
import path   from 'path';

// Load default .env
dotenv.config();

// Load environment-specific file
const env = process.env.TEST_ENV || 'qa';
dotenv.config({ path: path.resolve(`.env.${env}`) });

// ⚠️ process.env values are ALWAYS strings!
const port    = parseInt(process.env.DB_PORT || '5432');  // string → number
const debug   = process.env.DEBUG === 'true';             // string → boolean
const baseUrl = process.env.BASE_URL || 'http://localhost:3000';

TS-02

Variables — var, let, const

⚠️
Never use var in modern TypeScript/JavaScript. It has function scope, not block scope, and causes bugs. Always use const or let.
typescript — var vs let vs const
// ── var ── (AVOID — legacy, function-scoped, hoisted)
var x = 10;
if (true) {
  var x = 99;   // ← SAME x! leaks out of block — BUG RISK
}
console.log(x); // 99 — surprise!

// ── let ── (block-scoped, can reassign)
let count = 0;
count = 1;         // ✅ reassign ok
if (true) {
  let count = 99;  // different count — block scoped
}
console.log(count); // 1 — correct!

// ── const ── (block-scoped, CANNOT reassign)
const name = 'Sumit';
// name = 'Other';  ← ❌ TypeScript ERROR at compile time

// ⚠️ const object properties CAN change (reference is const, not content)
const user = { name: 'Sumit', age: 30 };
user.age = 31;     // ✅ property change ok
// user = {};       ← ❌ can't reassign the whole object

TS-08

type vs interface — When to Use Which

interface — open, extendable
interface Animal {
  name:  string;
  sound: string;
}

// extend interface
interface Dog extends Animal {
  breed: string;
}

// merge (declaration merging)
interface Animal {
  age: number;   // ← added to original!
}

// implements in class
class Cat implements Animal {
  name = 'Cat';
  sound = 'Meow';
}
type — sealed, flexible
type Animal = {
  name:  string;
  sound: string;
};

// extend type using & (intersection)
type Dog = Animal & {
  breed: string;
};

// type CAN do things interface can't:
type ID = string | number;   // union
type Tuple = [string, number]; // tuple
type Status = 'on' | 'off';   // literal
type Fn = (x: number) => void; // function shape

// can also implement in class
class Cat implements Animal {
  name = 'Cat';
  sound = 'Meow';
}
🧠 Simple Rule
Use interface when: defining shape of a class OR an object that might be extended Use type when: union types, tuples, aliases, function signatures, complex combinations In practice for Playwright: interface → page objects, fixture types, API response shapes type → union of string literals, Record aliases, complex generics

Modules — import/export & Same Variable Names in Different Files

💡
In Java, every file is a class and package system handles namespacing. In TypeScript, every file is a module — variables are private to the file by default. You must explicitly export to share and import to use. No conflict between same-named variables in different files.

export abstract class BasePage {

    return this.page.title();
  }

    await this.page.screenshot({ path: `screenshots/${name}.png` });
  }
}

// pages/LoginPage.ts
import { Page }     from '@playwright/test';
import { BasePage } from './BasePage';

export class LoginPage extends BasePage {

  constructor(page: Page) { super(page); }

  async open(): Promise<void> {
    await this.page.goto('/login');
  }

  async login(email: string, password: string): Promise<void> {
    await this.page.getByLabel('Email').fill(email);
    await this.page.getByLabel('Password').fill(password);
    await this.page.getByRole('button', { name: 'Login' }).click();
  }

  async getErrorMessage(): Promise<string> {
    return this.page.getByTestId('error-msg').innerText();
  }
}

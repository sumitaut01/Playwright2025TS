TypeScript Project Config

tsconfig.json

The TypeScript compiler's instruction manual. Tells tsc how strict to be, what files to include, and where to output. Every Playwright+TS project needs one.

What is it exactly?

tsconfig.json is to TypeScript what pom.xml is to Maven — it configures the build tool. When you run npx playwright test, Playwright's test runner uses ts-jest / esbuild internally, which reads tsconfig.json to know how to transpile your .ts test files before running them.

Playwright's Default tsconfig.json

tsconfig.jsonJSON
{
  "compilerOptions": {

    // ── TARGET & MODULE ──────────────────────────────────────────
    "target": "ESNext",
    // What JS version to compile TO. ESNext = latest JS features preserved.
    // Node 18+ understands ESNext so no downgrade needed.

    "module": "commonjs",
    // Import system to use in output. CommonJS = require() style (Node default).
    // Use "ESNext" if you're on ESM / "type":"module" in package.json.

    "moduleResolution": "node",
    // How to resolve import paths. "node" = look in node_modules (standard).

    // ── LANGUAGE FEATURES ────────────────────────────────────────
    "lib": ["ESNext", "DOM"],
    // Which built-in type definitions to include.
    // DOM = browser globals (window, document, fetch) — needed for page evals.

    "experimentalDecorators": true,
    // Enables @Decorator syntax — used in some POM patterns.

    // ── STRICTNESS ───────────────────────────────────────────────
    "strict": true,
    // Umbrella flag that enables ALL strict checks below:
    //   strictNullChecks    → string | null must be handled
    //   strictFunctionTypes → function param types are checked
    //   noImplicitAny       → every var must have a known type
    //   strictPropertyInit  → class fields must be initialized

    "noUnusedLocals": true,
    // Error on declared-but-unused variables (catches dead code)

    "noUnusedParameters": true,
    // Error on function params that are declared but never used

    // ── OUTPUT ───────────────────────────────────────────────────
    "outDir": "./dist",
    // Where compiled JS files go. Playwright usually ignores this
    // (it compiles in-memory), but it's good practice to set it.

    "sourceMap": true,
    // Generates .map files so debugger shows TS source, not compiled JS.
    // Essential for VS Code breakpoints in test files.

    "declaration": true,
    // Generates .d.ts type definition files. Only needed if publishing a lib.

    // ── PATH ALIASES ─────────────────────────────────────────────
    "baseUrl": ".",
    // Root for resolving path aliases below. "." = project root.

    "paths": {
      "@pages/*": ["pages/*"],
      // import { LoginPage } from '@pages/LoginPage'
      // instead of '../../pages/LoginPage'
      "@fixtures/*": ["fixtures/*"],
      "@utils/*": ["utils/*"]
    }
  },

  // ── FILE INCLUSION ───────────────────────────────────────────────
  "include": [
    "tests/**/*.ts",    // all test files
    "pages/**/*.ts",    // Page Objects
    "fixtures/**/*.ts", // custom fixtures
    "utils/**/*.ts"     // helpers
  ],

  "exclude": [
    "node_modules",
    "dist"
  ]
}

Key Options Explained Visually

🎯
"strict": true
The single most important flag. Turns TypeScript from a "type-hint language" into a "type-safe language". Catches null pointer bugs at compile time — same safety as Java's type system.
🗺️
"paths" aliases
Replace ugly relative imports like ../../../../pages/LoginPage with clean @pages/LoginPage. Must also configure in playwright.config.ts via require('tsconfig-paths').
🐛
"sourceMap": true
Without this, VS Code breakpoints in .ts files won't work — the debugger can only see the compiled .js output. Always keep this true.
📦
"target" vs "module"
target = JS version of your output. module = import system of your output. For Node.js Playwright tests: target: "ESNext", module: "commonjs" is the safe default.
📋
"include" / "exclude"
Controls which files TypeScript type-checks. If a file isn't in include, its types won't be verified and you'll get mystery errors at runtime.
🧩
"lib": ["DOM"]
Adds browser global types. Needed for page.evaluate(() => document.title) — without DOM in lib, TypeScript doesn't know what document is.

Multiple tsconfig files (common pattern)

Real-world projects often have 2–3 tsconfig files: a base, one for src, one for tests. Each extends the base and overrides only what differs.
tsconfig.base.json ← shared settingsJSON
{
  "compilerOptions": {
    "target": "ESNext",
    "strict": true,
    "sourceMap": true,
    "moduleResolution": "node"
  }
}
tsconfig.json ← Playwright test config (extends base)JSON
{
  "extends": "./tsconfig.base.json",
  // ↑ inherits all base settings, only override what's different
  "compilerOptions": {
    "module": "commonjs",
    "lib": ["ESNext", "DOM"],
    "baseUrl": ".",
    "paths": { "@pages/*": ["pages/*"] }
  },
  "include": ["tests/**/*.ts", "pages/**/*.ts", "fixtures/**/*.ts"]
}

Architecture

Understand how Browser → BrowserContext → Page stack works — isolated, parallel, powerful.

Object Hierarchy

Browser
← Chromium / Firefox / WebKit process
BrowserContext
← Isolated session (cookies, storage). Like Incognito window.
Page
← Single browser tab. Has its own URL, DOM.
Locator
← Lazy element reference. Re-queried on every action.
📌 Key Insight: Each test automatically gets its own BrowserContext. This means tests are fully isolated — no shared cookies, localStorage, or state between tests. Perfect for parallel execution.

architecture-demo.spec.ts TS 
import { test, expect } from '@playwright/test';

// Each test() gets a fresh Page (and BrowserContext) automatically
test('example', async ({ page }) => {
  // page = isolated tab in a fresh context
  await page.goto('/');
});

// Creating contexts manually (for multi-user scenarios)
test('two users', async ({ browser }) => {
  const adminCtx = await browser.newContext({
    storageState: 'auth/admin.json'
  });
  const userCtx = await browser.newContext({
    storageState: 'auth/user.json'
  });

  const adminPage = await adminCtx.newPage();
  const userPage = await userCtx.newPage();

  // Both running in parallel, completely isolated
  await adminPage.goto('/admin/dashboard');
  await userPage.goto('/user/home');
});

// === NAVIGATION ===
await page.goto('https://example.com');

👁️
Visibility Group
Is the element on screen? Can the user see it?
toBeVisible()
Element exists in DOM and is not hidden by CSS (display/visibility/opacity). Fails if element is display:none or visibility:hidden.
Use for: buttons, headings, cards, modals
toBeHidden()
Opposite of toBeVisible. Passes if element is hidden or does not exist at all in the DOM.
Use for: closed dialogs, hidden error messages
toBeInViewport()
Element is within the visible viewport area — useful for scroll tests. Not the same as toBeVisible() which ignores scroll position.
Use for: sticky headers, scroll-to-anchor tests
toHaveCount(n)
Matches exactly n elements. Use toHaveCount(0) as a clean way to assert an element does not exist — better than a try/catch.
Use for: list lengths, table rows, search results
visibility-group.spec.tsTS
const modal   = page.getByRole('dialog');
const banner  = page.getByTestId('error-banner');
const rows    = page.getByRole('row');

await expect(modal).toBeVisible();               // modal is open and on screen
await expect(banner).toBeHidden();               // no error right now
await expect(modal).toBeInViewport();            // not scrolled off-screen
await expect(rows).toHaveCount(30);              // exactly 30 pricing rules
await expect(page.getByRole('alert')).toHaveCount(0);  // no errors present
🔘
State Group
Is the element interactive? Ticked? Editable?
toBeEnabled()
Element does not have the disabled attribute. Safe to interact with.
toBeDisabled()
Element has disabled attribute. Submit button before form is valid, locked fields, etc.
toBeChecked()
Checkbox or radio is in checked state. Pass { checked: false } to assert unchecked.
toBeEditable()
Input/textarea is editable — not readonly or disabled. User can type into it.
toBeFocused()
Element has keyboard focus. Useful for tab-order and accessibility tests.

await expect(page.getByLabel('Remember me')).toBeChecked();

await expect(page.getByLabel('Email')).toBeEditable();

📝
Text & Value Group
What does the element say? What's typed in it? Which option is selected?
⚡ The Most Important Distinction
toHaveText('Dashboard')
Matches the full trimmed text. If the heading says "My Dashboard", this FAILS. Also accepts regex: /Dashboard/i
toContainText('Dashboard')
Just checks the text is somewhere inside the element. "My Dashboard" PASSES. Safer for dynamic content.
toHaveValue()
Current value of <input>, <textarea>, or <select>. What the user sees typed/selected.
toHaveValues()
For multi-select — array of selected values. Distinct from singular toHaveValue().
toHaveInputType()
Asserts the type attribute of an input: 'email', 'password', 'number' etc.
toHaveText([…])
Pass an array to match text of multiple elements at once — validates all list items or table cells in one shot.
text-value-group.spec.tsTS
await expect(page.getByRole('heading')).toHaveText('Dashboard');
await expect(page.getByRole('heading')).toHaveText(/dashboard/i);  // regex
await expect(page.getByRole('paragraph')).toContainText('Welcome');

await expect(page.getByLabel('Email')).toHaveValue('test@x4v.com');
await expect(page.getByLabel('Tags')).toHaveValues(['ev', 'solar']);

// Match all list items at once (order matters!)
await expect(page.getByRole('listitem')).toHaveText([
  'Rule A', 'Rule B', 'Rule C'
]);

await expect(page.getByLabel('Password')).toHaveInputType('password');
🏷️
DOM & Style Group
HTML attributes, CSS classes, computed styles, ARIA roles.
toHaveAttribute(name, val)
Checks an HTML attribute. toHaveAttribute('href', '/login'). Pass only name to check existence.
toHaveClass(className)
Checks the element has a CSS class. Pass a string or regex. Does not require exact class list match — just that the class is present.
toHaveCSS(prop, val)
Checks computed CSS — not the class, but what the browser actually renders. Values must be in computed form: 'rgb(0,0,0)' not 'black'.
toHaveId(id)
Shorthand for toHaveAttribute('id', value). Cleaner when you specifically care about the id attribute.
toHaveRole(role)
Checks the computed ARIA role. Useful for accessibility testing — ensures your Angular components expose the right semantic role to screen readers.
dom-style-group.spec.tsTS
const link = page.getByRole('link', { name: 'Pricing' });
const btn  = page.getByRole('button', { name: 'Save' });

await expect(link).toHaveAttribute('href', '/pricing');
await expect(btn).toHaveClass('primary-btn');
await expect(btn).toHaveCSS('background-color', 'rgb(0, 123, 255)');
await expect(page.getByTestId('logo')).toHaveId('company-logo');
await expect(page.getByTestId('alert-box')).toHaveRole('alert');
🌐
Page-Level Group
Assert on the page itself, not a locator. URL, title, screenshot.
toHaveURL(url)
Asserts on page directly. Accepts string, glob ('**/dashboard'), regex, or predicate function.
toHaveTitle(title)
Checks <title> tag text. Accepts string or regex. Great for quick smoke tests verifying you're on the right page.
toHaveScreenshot()
Visual regression — pixel-by-pixel compare with a saved golden screenshot. First run creates the golden file. Subsequent runs compare against it.
expect.soft()
Not a matcher but a modifier. expect.soft(locator) records failures but doesn't stop the test. All soft failures are reported together at end.
page-level-group.spec.tsTS
await expect(page).toHaveURL('https://app.x4v.com/dashboard');
await expect(page).toHaveURL(/dashboard/);               // regex
await expect(page).toHaveURL(url => url.includes('dash')); // predicate
await expect(page).toHaveTitle('X4V Unite · Dashboard');

// Visual regression
await expect(page).toHaveScreenshot('dashboard.png');
await expect(page.getByTestId('chart')).toHaveScreenshot();

// Soft assertions — all collected, test continues after each
await expect.soft(page).toHaveURL(/dashboard/);
await expect.soft(page.getByRole('heading')).toHaveText('Dashboard');
await expect.soft(page.getByTestId('user-name')).toHaveText('Sumit');
// 👆 all 3 checked regardless of individual pass/fail
🔁 Negate any matcher with .not
await expect(btn).not.toBeVisible();
await expect(page).not.toHaveURL(/login/);
Works on every single matcher — just prefix with .not
⏱️ All assertions auto-retry
Every expect() here polls until true or timeout (default 5s). No waitFor wrappers needed. Set per-assertion: toBeVisible({ timeout: 15000 })
Timing · Lazy Loading · Auto-wait

Waits & Auto-wait

Playwright eliminates most explicit waits through smart auto-waiting and lazy locators. Understand when it helps, when it doesn't, and what to do in each case.

Explicit Waits
Lazy Loading & Auto-wait
Golden Rule: If you find yourself writing page.waitForTimeout(3000), that's a code smell. Use event-based waits instead.
waits.spec.tsTS
// ✅ Wait for navigation
await page.waitForURL('**/dashboard');
await page.waitForURL(/dashboard/);

// ✅ Wait for network
await page.waitForLoadState('networkidle');    // No requests for 500ms
await page.waitForLoadState('domcontentloaded');

// ✅ Wait for element state
await page.getByRole('dialog').waitFor();
await page.getByRole('dialog').waitFor({ state: 'visible' });
await page.getByRole('spinner').waitFor({ state: 'hidden' });
// state options: 'attached' | 'detached' | 'visible' | 'hidden'

// ✅ Wait for response (use Promise.all pattern — see Promise.all section)
const [response] = await Promise.all([
  page.waitForResponse('**/api/pricing/**'),
  page.getByRole('button', { name: 'Calculate' }).click(),
]);
expect(response.status()).toBe(200);

// ✅ Wait for custom JS condition
await page.waitForFunction(() =>
  document.querySelectorAll('.price-row').length > 5
);

// ❌ Avoid — hides real issues, makes CI slow
await page.waitForTimeout(3000);

waitFor state values

StateMeaningCommon Use
'visible'In DOM and not hidden by CSSWait for modal to open, content to appear
'hidden'Not visible (or not in DOM at all)Wait for spinner to disappear
'attached'Present in DOM (even if hidden)Wait for element to be inserted
'detached'Removed from DOM entirelyWait for deleted row to be gone

Part 1 — Lazy Locators

A locator is not an element. It's a recipe — a description of how to find an element. The actual DOM query only fires when you perform an action on it.

SELENIUM (JAVA)
WebElement el =
  driver.findElement(
    By.id("submit"));
// ← DOM queried NOW
// el holds a live DOM node
el.click();
// if DOM rebuilt → 💥 Stale
→
PLAYWRIGHT (TS)
const btn =
  page.getByRole('button');
// ← nothing queried yet
// btn is just a recipe 📋
await btn.click();
// DOM queried HERE ✅
lazy-locators-explained.tsTS
// These 3 lines query NOTHING — they just build a recipe
const submitBtn  = page.getByRole('button', { name: 'Submit' });
const emailInput = page.getByLabel('Email');
const errorMsg   = page.getByTestId('error-banner');
// Zero DOM access so far. No cost. No risk.

// DOM is queried HERE — fresh every time
await emailInput.fill('test@x4v.com'); // query #1: find label "Email"
await submitBtn.click();               // query #2: find button "Submit"
await expect(errorMsg).toBeHidden();  // query #3: find error-banner

// Safe to reuse the same locator after page changes:
await page.reload();                   // DOM rebuilt
await submitBtn.click();               // ✅ re-queries fresh DOM — no stale issue

// Even in a loop — each iteration gets the freshest DOM
for (let i = 0; i < 3; i++) {
  await page.getByRole('button', { name: 'Add Row' }).click();
  // Each iteration: fresh DOM query, fresh click
}
⚠️ One thing that IS NOT lazy: page.$() and page.$$() return ElementHandle / ElementHandle[] — these hold a direct DOM reference and CAN become stale. They're the old API. Never use them in new tests — always use page.locator() / getBy*().

Part 4 — Where Auto-wait Does NOT Help ⚠️

Auto-wait only checks the element you're acting on. It has no knowledge of background work, polling intervals, or content that changes after the element becomes visible.
autowait-limits.spec.tsTS
// ❌ 1. Lazy-loaded images / infinite scroll
// Images below the fold are not loaded until scrolled into view.
// Auto-wait won't scroll for you.
await page.getByRole('img', { name: 'Rule diagram' }).scrollIntoViewIfNeeded();
await expect(page.getByRole('img')).toHaveAttribute('src', /diagram/);
// ✅ Fix: scroll first, THEN assert

// ❌ 2. Content that changes AFTER becoming visible
// A counter that ticks from 0 → 30 over 2 seconds.
// The <span> is visible from the start, but value changes later.
await expect(page.getByTestId('rule-count')).toHaveText('30');
// ✅ This DOES work — toHaveText() retries until text = "30" ✅
// (assertions auto-retry independently — this is fine)

// ❌ 3. Background WebSocket / polling data
// A live price feed updates a cell every 5s via WebSocket.
// Auto-wait doesn't know about WebSocket messages.
await page.waitForFunction(() => {
  const el = document.querySelector('[data-testid="live-price"]');
  return el && el.textContent !== '—';  // wait until value arrives
});
// ✅ Fix: waitForFunction with a custom condition

// ❌ 4. networkidle on pages with polling requests
// If your app polls /api/heartbeat every 2s, networkidle NEVER fires.
await page.waitForLoadState('networkidle'); // hangs forever on polling apps
// ✅ Fix: use domcontentloaded or wait for a specific element instead
await page.waitForLoadState('domcontentloaded');
await expect(page.getByTestId('dashboard-content')).toBeVisible();

// ❌ 5. File downloads
// Auto-wait can't know a file has finished downloading.
const [download] = await Promise.all([
  page.waitForEvent('download'),
  page.getByRole('button', { name: 'Export CSV' }).click(),
]);
await download.saveAs('/tmp/rules.csv');
// ✅ Fix: waitForEvent('download') + Promise.all pattern

Summary: Auto-wait decision table

ScenarioAuto-wait covers it?What to use
Element rendered after API call ✅ Yes Just act — Playwright retries until it appears
Button disabled until form valid ✅ Yes Just .click() — enabled check is built in
Element animating / transitioning ✅ Yes Stability check waits for animation to stop
Overlay covering element ✅ Yes "Receives events" check waits for overlay removal

For anything that happens after the element is ready — use expect() retries (for DOM content) or waitForFunction / waitForEvent / Promise.all (for everything else).

Page Object Model

Page Object Class

pages/LoginPage.tsTS
import { type Page, type Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;

  // Locators defined as class properties (lazy — not queried until used)
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitBtn: Locator;
  readonly errorMsg: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput    = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitBtn     = page.getByRole('button', { name: 'Sign In' });
    this.errorMsg      = page.getByRole('alert');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitBtn.click();
  }

  async loginAndWait(email: string, password: string) {
    await this.login(email, password);
    await this.page.waitForURL('**/dashboard');
  }
}
💡 Key difference from Selenium: Locators in Playwright are lazy — assigning this.submitBtn = page.getByRole(...) does NOT touch the DOM. The DOM query only fires when you call .click(), .fill(), etc. No stale element exceptions!

TypeScript / JavaScript Concept

The ... Spread Operator

Used heavily in Playwright configs, fixtures, and POM. Understand it once and configs become obvious.

What does ... do?

The spread operator expands an object or array into individual pieces — like pouring a glass of water into a larger container.

📤
Object Spread
Copies all key-value pairs from one object into another. {...a, ...b} merges both — b wins on conflicts.
📋
Array Spread
Expands array items inline: [...arr1, ...arr2] concatenates. [...arr] shallow-clones.
📥
Rest (opposite)
Same syntax in destructuring means "collect the rest": const { a, ...rest } = obj — rest has everything except a.

Core Mechanics

spread-basics.tsTS
// ── Object spread ───────────────────────────────────────────────
const base = { timeout: 30000, headless: true };
const override = { headless: false, slowMo: 200 };

const merged = { ...base, ...override };
// → { timeout: 30000, headless: false, slowMo: 200 }
//   ↑ base props copied    ↑ override wins on 'headless'

// ── Shallow clone (no mutation) ──────────────────────────────────
const original = { user: 'sumit', role: 'qa' };
const clone = { ...original };        // new object, same values
clone.role = 'admin';               // original.role is still 'qa'

// ── Add/override single property ────────────────────────────────
const withExtra = { ...original, role: 'admin', verified: true };
// → { user: 'sumit', role: 'admin', verified: true }

// ── Array spread ─────────────────────────────────────────────────
const browsers = ['chromium', 'firefox'];
const all = [...browsers, 'webkit'];   // ['chromium', 'firefox', 'webkit']

// ── Rest (collecting remaining props) ───────────────────────────
const config = { baseURL: 'http://localhost', timeout: 5000, headless: true };
const { baseURL, ...rest } = config;
// baseURL = 'http://localhost'
// rest    = { timeout: 5000, headless: true }

Java vs TypeScript mental model

Java (what you know)TypeScript ... equivalent
new HashMap(existingMap){...existingObj} — shallow copy
map.putAll(otherMap){...map1, ...map2} — merge, last wins
new ArrayList(list)[...list] — array copy
Collections.addAll(a, b)[...a, ...b] — concatenate
No direct equivalentconst {x, ...rest} = obj — destructure + collect rest
💡 Key rule: Later properties win. { timeout: 5000, ...override } — override can change timeout. { ...override, timeout: 5000 } — your literal 5000 always wins regardless of override.
Fixture System Deep Dive

base.extend<T>

The engine behind all custom fixtures. Understand this pattern and you can build any test infrastructure Playwright is capable of.

The Signature Explained

anatomy.tsTS
// Full signature:
const test = base.extend<YourFixtureTypes>({
  fixtureName: async ({ /* other fixtures to inject */ }, use) => {
    // SETUP: runs before the test (like @BeforeMethod in TestNG)
    const value = await setupSomething();

    await use(value);  // ← HAND CONTROL to the test

    // TEARDOWN: runs after the test (like @AfterMethod)
    await cleanupSomething(value);
  },
});
🔤
<T> Generic Type
Tells TypeScript what custom fixtures you're adding. Autocomplete in tests works because of this. T is a plain object type with fixture names as keys.
📌
base Parameter
base is the imported test renamed. It carries all built-in fixtures. Your extension wraps it — you get page, request, etc. for free.
🔁
use(value) Pivot
Everything before await use() is setup. Everything after is teardown. Playwright guarantees teardown runs even if the test throws.
🔗
DI / Injection
Fixtures can depend on other fixtures. Playwright resolves the dependency graph automatically — just declare what you need in the destructured argument.

Async JavaScript / TypeScript

Promise.all

Run multiple async operations concurrently instead of sequentially. Critical for Playwright's waitForResponse + click pattern — and for understanding async TypeScript in general.

What is a Promise?

Before Promise.all, understand what a Promise actually is.

Promise = a placeholder for a future value. When you call an async function, it immediately returns a Promise object — a "ticket" saying "the result isn't ready yet, but I'll fill it in." The actual value arrives later when the operation completes.
promise-basics.tsTS
// A Promise has 3 states:
// ⏳ Pending   → operation still running
// ✅ Fulfilled → completed successfully, holds a value
// ❌ Rejected  → failed, holds an error

// page.goto() returns a Promise<Response>
const promise: Promise<Response> = page.goto('/');
// At this line the navigation has STARTED but not finished

const response = await promise;
// await PAUSES this function until the promise settles
// response now holds the actual Response value

// Shorthand (what you normally write):
const res = await page.goto('/');  // await + call in one line

The Sequential Problem

⚠️ Classic Playwright bug: If you await click() first and then call waitForResponse(), the response may have already arrived before you started waiting — and you'll miss it, causing a timeout.
sequential-problem.spec.tsTS
// ❌ WRONG — Race condition. Response may arrive before we start listening.
test('broken: click then wait', async ({ page }) => {
  await page.getByRole('button', { name: 'Load Rules' }).click();
  // ↑ click fires the request immediately

  const response = await page.waitForResponse('**/api/rules');
  // ↑ we start listening AFTER the response may have already come back
  // If the API is fast → timeout! 💥
});

// ❌ ALSO WRONG — awaiting sequentially wastes time
test('slow: sequential awaits', async ({ page }) => {
  await page.goto('/page-a');   // waits for A to finish
  await page.goto('/page-b');   // THEN starts B — total: A + B seconds
});

Promise.all — Run in Parallel

Promise.all([p1, p2, p3]) starts ALL promises simultaneously and waits until every single one is fulfilled. Total time = slowest operation (not sum of all).

promise-all-basics.tsTS
// Signature:
// Promise.all(iterable): Promise<Array>
// → takes array of Promises
// → returns ONE Promise that resolves when ALL resolve
// → resolves to an array of results, in the same order as input
// → REJECTS immediately if ANY promise rejects (fail-fast)

// Time comparison:
// Sequential: await task1()  →  await task2()  →  3s + 4s = 7s total
// Parallel:   Promise.all([task1(), task2()])   →  max(3s, 4s) = 4s total

const [result1, result2, result3] = await Promise.all([
  fetchUsers(),       // starts immediately
  fetchProducts(),    // starts immediately (not waiting for fetchUsers)
  fetchPricing(),     // starts immediately
]);
// All three run concurrently. Destructuring maps results by position.

The #1 Playwright Use Case — Click + WaitForResponse

Golden pattern: Register the listener BEFORE triggering the action. Promise.all starts both "listening" and "clicking" at the same instant — so no response can be missed.
click-and-wait.spec.tsTS
// ✅ CORRECT — Promise.all fires both simultaneously
test('click triggers API call and UI updates', async ({ page }) => {
  await page.goto('/pricing/rules');

  const [response] = await Promise.all([
    page.waitForResponse('**/api/pricing/rules'),
    // ↑ starts LISTENING now (not awaited yet)

    page.getByRole('button', { name: 'Load Rules' }).click(),
    // ↑ fires the click now — which triggers the network request
  ]);
  // Playwright.all waits for BOTH: response received AND click done

  expect(response.status()).toBe(200);
  const body = await response.json();
  expect(body.rules).toHaveLength(30);
});

// ✅ Wait for navigation + click simultaneously
test('click navigates to dashboard', async ({ page }) => {
  await Promise.all([
    page.waitForURL('**/dashboard'),
    page.getByRole('button', { name: 'Go to Dashboard' }).click(),
  ]);
  // URL is guaranteed to be /dashboard at this point
  await expect(page).toHaveURL(/dashboard/);
});

// ── Promise.all ───────────────────────────────────────────────────
// Waits for ALL. Fails fast if ANY rejects.
const [a, b] = await Promise.all([task1(), task2()]);

// ── Promise.allSettled ────────────────────────────────────────────
// Waits for ALL regardless of failure. Never throws.
// Returns array of { status: 'fulfilled'|'rejected', value|reason }
const results = await Promise.allSettled([
  request.get('/api/rules'),
  request.get('/api/might-fail'),
]);
results.forEach(r => {
  if (r.status === 'fulfilled') console.log(r.value);
  else console.error('Failed:', r.reason);
});

// ── Promise.race ─────────────────────────────────────────────────
// Resolves/rejects as soon as the FIRST promise settles (any outcome)
// Use case: timeout race
const winner = await Promise.race([
  page.waitForSelector('.success-banner'),
  page.waitForSelector('.error-banner'),
]);
// Whichever banner appears first wins

// ── Promise.any ──────────────────────────────────────────────────
// Resolves as soon as FIRST succeeds. Rejects only if ALL fail.
const first = await Promise.any([
  tryPrimaryEndpoint(),
  tryFallbackEndpoint(),
]);

Common Mistakes

mistakes.spec.tsTS
// ❌ Mistake 1: Forgetting await on Promise.all
Promise.all([task1(), task2()]);  // fires and forgets — test won't wait!
// ✅ Fix:
await Promise.all([task1(), task2()]);

// ❌ Mistake 2: Awaiting inside the array (defeats the purpose)
await Promise.all([
  await task1(),   // ← await resolves BEFORE Promise.all sees it
  await task2(),   // ← now sequential again!
]);
// ✅ Fix: pass the Promise, not the resolved value
await Promise.all([task1(), task2()]);

// ❌ Mistake 3: One rejection kills everything silently
await Promise.all([
  successfulTask(),
  mightFailTask(),    // if this rejects, entire Promise.all rejects
  anotherTask(),      // this result is lost even if it succeeded
]);
// ✅ Fix: use allSettled if partial success is acceptable
await Promise.allSettled([successfulTask(), mightFailTask(), anotherTask()]);
🔑 One-line summary: Use Promise.all whenever you have two or more async operations that don't depend on each other — especially the waitForResponse / waitForURL + click pattern which is required in virtually every real Playwright test suite.

Element Not Found

You called an action or assertion on a locator that matched zero elements in the DOM.

Error message you'll see:
TimeoutError: locator.click: Timeout 30000ms exceeded.
=========================== logs ===========================
waiting for getByRole('button', { name: 'Submit' })
============================================================
📌 Key insight: Playwright does NOT throw an "element not found" immediately. Because of auto-wait, it keeps retrying for the full timeout (default 30s) hoping the element will appear. The error you get is always a TimeoutError — but the root cause is "element never showed up."
not-found-causes.spec.tsTS
// ❌ Scenario 1: Wrong role/name
await page.getByRole('button', { name: 'Submit' }).click();
// But the button's text is actually 'Save' → TimeoutError after 30s

// ❌ Scenario 2: Element inside a different frame
await page.getByLabel('Card Number').fill('4111...');
// Input lives inside an iframe → needs frameLocator() first

// ❌ Scenario 3: Querying before page finishes loading
await page.goto('/dashboard');
await page.getByTestId('rules-table').click();
// Table rendered by Angular after an API call — needs no extra wait
// because auto-wait handles it, BUT only if you have enough timeout

// ✅ Check if element exists WITHOUT timing out:
const count = await page.getByRole('button', { name: 'Submit' }).count();
if (count > 0) {
  await page.getByRole('button', { name: 'Submit' }).click();
}

// ✅ Check visibility without throwing:
const isVisible = await page.getByRole('dialog').isVisible();
// Returns true/false — never throws even if element absent

// ✅ Soft assert (doesn't stop the test):
await expect.soft(page.getByText('Welcome')).toBeVisible();

// ✅ Assert element does NOT exist:
await expect(page.getByRole('dialog')).toHaveCount(0);
// or:
await expect(page.getByRole('dialog')).toBeHidden();

// ── Timeout hierarchy (each can be set independently) ────────────

// 1. Global test timeout (playwright.config.ts)
defineConfig({ timeout: 30000 });  // default 30s per test

// 2. expect/assertion timeout (polling timeout)
defineConfig({ expect: { timeout: 5000 } });  // default 5s

// 3. Per-action timeout (overrides global for one action)
await page.getByRole('button').click({ timeout: 10000 });

await expect(page.getByRole('table')).toBeVisible({ timeout: 15000 });

});

Multiple Elements Matching — Strict Mode

Playwright's locators are strict by default. If your locator matches more than one element and you perform a single-element action, it throws immediately — no retry, no timeout.

Error message:
Error: strict mode violation: getByRole('button', { name: 'Edit' }) resolved to 5 elements:
1) <button class="edit-btn">Edit</button> at index 0
2) <button class="edit-btn">Edit</button> at index 1
...
📌 Why strict mode is a feature, not a bug: In Selenium, findElement() silently picks the first match. You might be clicking the wrong element and not know it for weeks. Playwright's strict mode fails loudly, forcing you to be precise.
strict-mode.spec.tsTS
// ❌ Multiple <button>Edit</button> on the page → strict violation
await page.getByRole('button', { name: 'Edit' }).click();

// ✅ Fix 1: Scope the locator to a specific container
const firstRow = page.getByTestId('rule-row').first();
await firstRow.getByRole('button', { name: 'Edit' }).click();

// ✅ Fix 2: Use .nth() to select by index
await page.getByRole('button', { name: 'Edit' }).nth(0).click();

// ✅ Fix 3: Filter to a more specific match
await page.getByRole('button', { name: 'Edit' })
  .filter({ hasText: 'Edit Rule A' })
  .click();

// ✅ Fix 4: Use a parent locator to narrow scope
await page.locator('tr')
  .filter({ hasText: 'Power Rank 15' })
  .getByRole('button', { name: 'Edit' })
  .click();

// ✅ Intentionally working with ALL matches (not strict):
const allEditBtns = page.getByRole('button', { name: 'Edit' });
await expect(allEditBtns).toHaveCount(5);   // multi-element assertion ✅
const all = await allEditBtns.all();       // returns Locator[] ✅
for (const btn of all) {
  await expect(btn).toBeEnabled();
}

Page Reload & DOM Changes — No StaleElementException!

This is one of the biggest improvements over Selenium. Playwright locators are lazy and re-queried on every action — there is no stale element reference.

✅ In Playwright: There is no StaleElementReferenceException. A Locator object is not tied to a specific DOM node — it's a recipe for finding an element. Every time you act on it, it re-executes the query against the current DOM.
stale-element-comparison.spec.tsTS
// ── Selenium (Java) — StaleElementReferenceException ─────────────
// WebElement el = driver.findElement(By.id("submit"));
// driver.navigate().refresh();  // DOM rebuilt
// el.click();  // 💥 StaleElementReferenceException — el points to dead node

// ── Playwright — No such problem ─────────────────────────────────
const btn = page.getByRole('button', { name: 'Submit' });
await page.reload();   // page fully reloads, DOM rebuilt
await btn.click();    // ✅ Works fine — btn re-queries the DOM right now

// After SPA navigation (React/Angular route change):
const heading = page.getByRole('heading');
await page.getByRole('link', { name: 'Dashboard' }).click();
// Angular re-renders the view
await expect(heading).toHaveText('Dashboard');  // ✅ re-queries, finds new heading

// After partial re-render (list refreshed by an API poll):
const rows = page.getByRole('row');
await page.waitForResponse('**/api/rules');  // list refreshed
await expect(rows).toHaveCount(30);  // ✅ fresh query, no stale issue
⚠️ The one exception — ElementHandle (avoid it): If you use the low-level page.$('selector') API (returns an ElementHandle), you CAN get a detached node error because ElementHandle does hold a direct DOM reference. Stick to Locator and you'll never see this.
element-handle-warning.tsTS
// ❌ ElementHandle — old API, avoid in modern Playwright
const el = await page.$('button.submit');  // returns ElementHandle | null
await page.reload();
await el?.click();  // ⚠️ May throw — el is a direct DOM reference

// ✅ Locator — modern API, always use this
const locator = page.locator('button.submit');  // Locator object (lazy)
await page.reload();
await locator.click();  // ✅ Re-queries DOM, always safe

Syntax Errors & Missing await

The trickiest bugs — no red error in the terminal, test passes, but nothing actually happened.

⚠️ TypeScript won't always catch missing await. The code compiles fine. The test might even pass — because the assertion runs before the action completes, and the DOM happened to already be in the right state.
missing-await.spec.tsTS
// ── Missing await on action ───────────────────────────────────────
page.goto('/');          // ❌ navigation started but not awaited
page.click('button');   // ❌ fires immediately, might click on loading page
// Both return Promises that are just dropped. No error shown.
// ✅ Fix: always await async calls
await page.goto('/');
await page.getByRole('button').click();

// ── Missing await on assertion ────────────────────────────────────
expect(page.getByRole('heading')).toHaveText('Dashboard');
// ❌ Returns a Promise. Test PASSES even if heading is wrong!
// ✅ Fix:
await expect(page.getByRole('heading')).toHaveText('Dashboard');

// ── Missing parentheses on method call ────────────────────────────
await page.reload;    // ❌ page.reload is a function reference, NOT a call
// No error thrown — await on a non-Promise just resolves instantly
// Page was never actually reloaded. Test silently does nothing.
// ✅ Fix: add the parentheses
await page.reload();  // ✅ now actually calls the function

// ── Same issue with other methods ────────────────────────────────
await page.goBack;           // ❌ method reference, not called
await locator.click;         // ❌ same — no () = no action
await context.storageState; // ❌ no-op
// ✅ Always check: does the method have () after it?

// ── How TypeScript/ESLint can help ────────────────────────────────
// Add to your eslint config to catch floating promises:
// "@typescript-eslint/no-floating-promises": "error"
// "@typescript-eslint/no-misused-promises": "error"
// These rules make missing await a compile-time error ✅
💡 VS Code tip: Install the ESLint extension and enable @typescript-eslint/no-floating-promises. VS Code will underline any unawaited Promise in red — same as a syntax error. This catches the page.reload vs page.reload() class of bug automatically.
.eslintrc.js (add to your project)JS
module.exports = {
  parser: '@typescript-eslint/parser',
  plugins: ['@typescript-eslint'],
  rules: {
    '@typescript-eslint/no-floating-promises': 'error',
    '@typescript-eslint/no-misused-promises': 'error',
    '@typescript-eslint/await-thenable': 'error',
    // ↑ warns when you await a non-Promise (e.g. await "string")
  },
};

other-errors.spec.tsTS
// ── Error: Target closed ──────────────────────────────────────────
// 💥 Error: Target page, context or browser has been closed
// Cause: you closed the browser/context but kept using the page ref
await browser.close();
await page.goto('/');  // 💥 browser is closed
// Fix: close only in afterAll/fixture teardown, not mid-test

// ── Error: Frame detached ─────────────────────────────────────────
// 💥 Error: Frame was detached
// Cause: iframe was removed from DOM while you held a frame locator
// Fix: re-acquire frameLocator() after navigation
await page.goto('/payment');
const frame = page.frameLocator('#stripe-iframe');  // re-fetch each time
await frame.getByLabel('Card number').fill('4111111111111111');

// ── Error: execSync / context isolation violation ─────────────────
// 💥 Error: browserContext.storageState() cannot be called while test is running
// Cause: calling context-level APIs from the wrong scope
// Fix: call storageState in global setup, not inside a test

// ── Error: expect received undefined ─────────────────────────────
const text = await page.getByRole('heading').textContent();
expect(text).toBe('Dashboard');
// textContent() returns string | null — if null, expect fails weirdly
// ✅ Fix: use toHaveText() which handles null safely
await expect(page.getByRole('heading')).toHaveText('Dashboard');

// ── Error: page.locator() with empty selector ─────────────────────
page.locator('');
// 💥 Error: selector cannot be empty

// ── Error: Can't use locator inside waitForFunction ───────────────
await page.waitForFunction('locator(".x").isVisible()');
// 💥 locator() doesn't exist in browser context
// Fix: use pure JS inside waitForFunction
await page.waitForFunction(() =>
  document.querySelector('.x') !== null
);

01 / Fundamentals
Why does async even exist?

JavaScript runs on a single thread. If everything was synchronous, one slow operation would freeze the entire program. Async solves this — understand the "why" before the "how".

The Single-Thread Problem

JavaScript can only do one thing at a time. Imagine you're a chef (JavaScript) — you can only stand at one stove at a time. The question is: while waiting for water to boil, do you stand and stare... or do something else?

❌ Synchronous (Blocking)
// Waiter takes order → goes to kitchen
// → stands there waiting → comes back
// Other customers starve meanwhile!

fetchUserFromDB("user1");  // 3 seconds
fetchUserFromDB("user2");  // 3 seconds
// Total: 6 seconds (bad!)
✅ Asynchronous (Non-Blocking)
// Waiter takes orders from everyone
// → kitchen works in parallel
// → waiter delivers when ready

fetchUserFromDB("user1");  // started
fetchUserFromDB("user2");  // started
// Total: ~3 seconds (good!)
The Event Loop — Heart of Async JS

This is the engine that makes async possible. JavaScript has one Call Stack (what's running now) and a Task Queue (what's waiting). The Event Loop's job: keep checking — if the stack is empty, push the next task from the queue.

When you do await fetch(...) in Playwright, JS says: "go do this network call, I'll continue when you're done." The Event Loop picks it up and resumes your test function once the response arrives.

02 / Core Concept
Promises — The Building Block

Before understanding async/await, you must understand Promises. async/await is just syntactic sugar built on top of them.

What is a Promise?

A Promise is an object that represents the eventual result of an async operation. Like an IOU note: "I don't have the result yet, but I promise I'll give it to you later."

⏳
Pending
Initial state. Async work is still running. Neither fulfilled nor rejected yet.
✅
Fulfilled
Operation completed successfully. The .then() handler fires with the result value.
❌
Rejected
Operation failed. The .catch() handler fires with the error reason.
⚠️
A Promise can only transition once — pending → fulfilled OR pending → rejected. Once settled, it never changes state.
Creating a Promise
// Manual Promise creation (you'll rarely do this in Playwright)
const myPromise = new Promise((resolve, reject) => {
  // Do some async work...
  const success = true;

  if (success) {
    resolve("Data loaded!");   // Fulfills → .then() fires
  } else {
    reject(new Error("Failed!"));  // Rejects  → .catch() fires
  }
});

// Consuming a Promise with .then() / .catch()
myPromise
  .then((result) => console.log("Success:", result))
  .catch((err)    => console.error("Error:", err.message))
  .finally(()      => console.log("Always runs!"));
Callback Hell vs Promise Chains

This is WHY Promises were invented — to escape the "pyramid of doom":

😱 Callback Hell
login(user, (err, token) => {
  if (err) throw err;
  getProfile(token, (err, profile) => {
    if (err) throw err;
    getOrders(profile, (err, orders) => {
      if (err) throw err;
      render(orders);  // nested!
    });
  });
});
✅ Promise Chain
login(user)
  .then((token)   => getProfile(token))
  .then((profile) => getOrders(profile))
  .then((orders)  => render(orders))
  .catch((err)    => console.error(err));
// Flat! Readable! One .catch handles all!
Promise Static Methods — The Power Tools
// Promise.all — Wait for ALL, fail if ANY fails
const [users, products, orders] = await Promise.all([
  fetchUsers(),
  fetchProducts(),
  fetchOrders()
]);

// Promise.allSettled — Wait for ALL, never fails
const results = await Promise.allSettled([
  riskyOp1(),
  riskyOp2()
]);
// [{status: "fulfilled", value: ...}, {status: "rejected", reason: ...}]

// Promise.race — First one wins (fulfilled OR rejected)
const fastest = await Promise.race([
  fetchFromServer1(),
  fetchFromServer2()
]);

// Promise.any — First FULFILLED wins (ignores rejections)
const firstSuccess = await Promise.any([
  tryAPI1(),
  tryAPI2(),
  tryAPI3()
]);

Promise.all vs allSettled — Real Playwright Examples

This is where juniors get confused most. Here's exactly when to pick each one:

❌ Promise.all — one fail = all fail
// Running 3 API health checks
// If /orders is down, you lose ALL results
const [users, orders, products] = await
  Promise.all([
    checkApi('/users'),     // ✅ ok
    checkApi('/orders'),    // 💥 throws!
    checkApi('/products')  // never runs
  ]);
// Entire Promise.all rejects
// You don't know if /products was ok!
✅ Promise.allSettled — always get all results
// Each result has status + value/reason
const results = await
  Promise.allSettled([
    checkApi('/users'),
    checkApi('/orders'),
    checkApi('/products')
  ]);

results.forEach((r, i) => {
  if (r.status === 'fulfilled')
    console.log(`API ${i} ✅`, r.value);
  else
    console.log(`API ${i} ❌`, r.reason);
});
🎭
Playwright real use: Running assertions on multiple elements in parallel — use Promise.all when you need all to pass, Promise.allSettled when you want a full health report of what passed and what failed.
// Playwright: check multiple UI elements at once
const checks = await Promise.allSettled([
  expect(page.locator('.header')).toBeVisible(),
  expect(page.locator('.cart-count')).toHaveText('3'),
  expect(page.locator('.user-name')).toBeVisible(),
]);
const failed = checks.filter(r => r.status === 'rejected');
if (failed.length) console.log(`${failed.length} checks failed`);
03 / The Modern Way
async / await — Write Async Like Sync

async/await is the modern syntax for working with Promises. It makes your async code look and behave more like synchronous code — much easier to read and reason about.

The Golden Rules

Rule 1: async marks a function

Add async before function keyword. This tells JS: "this function will do async work and will ALWAYS return a Promise."

async function myFunc() {
  return "hello";
}
// Returns Promise, not string!
// Same as: return Promise.resolve("hello")

// Arrow function style:
const myFunc = async () => {
  return "hello";
}

Rule 2: await pauses inside async

Use await ONLY inside an async function. It pauses execution until the Promise resolves and gives you the value directly.

async function getUser() {
  // await "unwraps" the Promise
  const response = await fetch("/api/user");
  const data     = await response.json();
  return data;
}
// No .then() chains needed!
🚨
Common Mistake: You CANNOT use await outside an async function (except at top-level in modern JS/TS modules). If you forget async, you'll get a SyntaxError.
Transforming Promises → async/await

See how async/await simplifies Promise chains:

Promise Chain Style
function loadUserData(userId) {
  return fetch(`/users/${userId}`)
    .then(res => res.json())
    .then(user => {
      return fetch(`/orders/${user.id}`)
        .then(res => res.json());
    })
    .catch(err => console.error(err));
}
async/await Style
async function loadUserData(userId) {
  try {
    const res1   = await fetch(`/users/${userId}`);
    const user   = await res1.json();
    const res2   = await fetch(`/orders/${user.id}`);
    const orders = await res2.json();
    return orders;
  } catch (err) {
    console.error(err);
  }
}
async/await with TypeScript types

TypeScript adds type safety to async functions. Playwright is fully typed — this is what you see in Playwright code:

// Return type annotation: Promise<T>
async function fetchUser(id: number): Promise<User> {
  const res  = await fetch(`/api/users/${id}`);
  const data = await res.json() as User;
  return data;
}

// Playwright test signature — async is REQUIRED here
test('user can login', async ({ page }) => {
  //                 ↑ async arrow function!
  await page.goto('https://example.com');
  await page.fill('#username', 'sumit');
  await page.click('button[type=submit]');

  // TypeScript knows page.title() returns Promise<string>
  const title: string = await page.title();
});
✅
When you hover over page.goto() in VS Code, it shows Promise<Response | null>. The await unwraps it to Response | null.
Sequential vs Parallel Execution

This is the #1 performance concept in async code:

🐌 Sequential (Slow)
async function slow() {
  // Each waits for previous
  // Total: 3s + 3s + 3s = 9s
  const a = await fetchA();
  const b = await fetchB();
  const c = await fetchC();
}
// Use when B depends on A's result
⚡ Parallel (Fast)
async function fast() {
  // All start at same time
  // Total: max(3s,3s,3s) = 3s!
  const [a, b, c] = await
    Promise.all([
      fetchA(),
      fetchB(),
      fetchC()
    ]);
}
💡
Use sequential when step B needs the result of step A. Use parallel (Promise.all) when operations are independent of each other.
04 / Resilience
Error Handling in Async Code

Async errors are trickier than sync errors. Unhandled rejections can silently crash your tests. Master these patterns.

try/catch — The Primary Pattern
async function loginTest(page) {
  try {
    await page.goto('https://myapp.com/login');
    await page.fill('#email', 'test@test.com');
    await page.click('[type=submit]');

    // This might throw if element not found
    await page.waitForSelector('.dashboard', { timeout: 5000 });

  } catch (error) {
    // Catches ALL async errors in the try block above
    console.error('Login test failed:', error.message);
    throw error;  // Re-throw to fail the test

  } finally {
    // Always runs — cleanup, screenshots, etc.
    await page.screenshot({ path: 'test-result.png' });
  }
}
Common Async Errors — Know Them All
// ❌ Forgetting await
const title = page.title();
console.log(title);
// Prints: Promise { <pending> } 😱

// ✅ With await
const title = await page.title();
console.log(title);
// Prints: "My App" ✅
// ❌ await in non-async function
function badFunc() {
  const data = await fetch('...');
  // SyntaxError: await only valid
  // in async function!
}

// ✅ Mark function as async
async function goodFunc() {
  const data = await fetch('...');
}
Timeout Handling Pattern
// Helper: wrap a promise with a timeout
function withTimeout<T>(promise: Promise<T>, ms: number): Promise<T> {
  const timeout = new Promise<never>((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  );
  return Promise.race([promise, timeout]);
}

// Usage
try {
  const data = await withTimeout(slowApiCall(), 5000);
} catch (e) {
  // Handles both API error AND timeout
  if (e.message.includes('Timed out')) {
    console.log('API was too slow!');
  }
}

// 🏆 Result wrapper pattern (advanced, clean)
async function safe<T>(p: Promise<T>): Promise<[Error | null, T | null]> {
  try {
    return [null, await p];
  } catch (e) {
    return [e as Error, null];
  }
}

// Usage — no try/catch needed at call site
const [err, data] = await safe(fetchUser(1));
if (err) console.log('Failed:', err.message);
else    console.log('Got:', data);
Playwright-Specific Errors — Read Them, Fix Them

These are the exact errors you will see in Playwright. Every one is an async error. Learn to recognise the message → know the fix:

ERROR TYPE 1 — TimeoutError
TimeoutError: page.click: Timeout 30000ms exceeded.
=================================================
waiting for locator('#submit-btn')...
Cause: Element not found or not clickable within timeout.
Fix: Wrong selector, page still loading, or element behind overlay.
// ❌ Causes timeout if element appears late
await page.click('#submit-btn');

// ✅ Wait for it first, with meaningful timeout
await page.locator('#submit-btn').waitFor({ state: 'visible', timeout: 10000 });
await page.click('#submit-btn');
ERROR TYPE 2 — Element not interactable
Error: page.fill: Element is not an <input>, <textarea> or [contenteditable]
element matching selector "#username" is a <div>
Cause: Selector matched wrong element or page has dynamic structure.
Fix: Be more specific with selector, or wait for correct state.
// Use more specific locator
await page.locator('input#username[type="text"]').fill('sumit');
ERROR TYPE 3 — Navigation / detached frame
Error: page.click: Target page, context or browser has been closed
Error: page.$: Execution context was destroyed, most likely because of a navigation
Cause: Page navigated while you were still holding a reference to the old context.
Fix: Re-query elements after navigation, don't cache locators across page loads.
// ❌ Stale reference after navigation
const btn = await page.locator('#btn');
await page.goto('/new-page');
await btn.click();  // 💥 old page context!

// ✅ Re-query after navigation
await page.goto('/new-page');
await page.locator('#btn').click();  // ✅ fresh query
ERROR TYPE 4 — Strict mode / multiple matches
Error: page.locator: strict mode violation: locator('.btn') resolved to 4 elements
Cause: Your selector matched more than one element. Playwright strict mode requires exactly one.
Fix: Use .first(), .nth(n), or a more specific selector.
// ✅ Target specific element
await page.locator('.btn').first().click();
await page.locator('.btn').nth(2).click();
await page.locator('form#login .btn[type=submit]').click();
05 / Real-world Patterns
Async Patterns You'll Actually Use

These are battle-tested patterns for real automation and API testing scenarios.

Retry Pattern

Retry a flaky async operation N times before giving up — essential for unstable APIs:

async function retry<T>(
  fn: () => Promise<T>,
  retries = 3,
  delay = 1000
): Promise<T> {
  for (let i = 0; i <= retries; i++) {
    try {
      return await fn();
    } catch (e) {
      if (i === retries) throw e;
      console.warn(`Attempt ${i+1} failed, retrying in ${delay}ms...`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error('unreachable');
}

// Usage
const data = await retry(() => fetch('/api/unstable').then(r => r.json()));
Polling Pattern — Wait until condition is true
async function pollUntil<T>(
  fn: () => Promise<T | null>,
  interval = 500,
  timeout  = 10000
): Promise<T> {
  const deadline = Date.now() + timeout;

  while (Date.now() < deadline) {
    const result = await fn();
    if (result !== null) return result;
    await new Promise(r => setTimeout(r, interval));
  }

  throw new Error(`Polling timed out after ${timeout}ms`);
}

// Wait until order status becomes "shipped"
const order = await pollUntil(async () => {
  const o = await fetchOrder(orderId);
  return o.status === 'shipped' ? o : null;
});
Concurrency Limiter — Don't overwhelm servers
async function batchProcess<T, R>(
  items: T[],
  fn: (item: T) => Promise<R>,
  batchSize = 5
): Promise<R[]> {
  const results: R[] = [];

  for (let i = 0; i < items.length; i += batchSize) {
    const batch = items.slice(i, i + batchSize);
    const batchResults = await Promise.all(batch.map(fn));
    results.push(...batchResults);
  }

  return results;
}

// Process 100 items, 5 at a time
const results = await batchProcess(
  userIds,
  (id) => fetchUser(id),
  5
);
async forEach vs for...of
❌ Broken Pattern
// forEach doesn't await! All fire at once
ids.forEach(async (id) => {
  const user = await fetchUser(id);
  // This doesn't execute sequentially!
  // forEach ignores the returned Promise
});
// Code here runs BEFORE all fetches done
✅ Correct Pattern
// for...of correctly awaits each
for (const id of ids) {
  const user = await fetchUser(id);
  // This IS sequential — one by one
  process(user);
}
// Code here runs AFTER all done ✅
🚨
Classic trap: array.forEach(async () => {}) does NOT await. Use for...of for sequential, or Promise.all(array.map(async () => {})) for parallel.

07 / Deep Dive
Advanced Async Concepts

Async generators, AbortController, and patterns for serious automation engineers.

Async Generators — Stream Data

Async generators produce values over time — useful for processing paginated API results:

// Async generator: yields paginated results one by one
async function* fetchAllPages(baseUrl: string) {
  let page = 1;

  while (true) {
    const res  = await fetch(`${baseUrl}?page=${page}`);
    const data = await res.json();

    yield data.items;  // yield each page's items

    if (!data.hasMore) break;
    page++;
  }
}

// Consume with for await...of
for await (const items of fetchAllPages('/api/products')) {
  console.log('Got page:', items.length, 'items');
}
AbortController — Cancel Async Operations
async function fetchWithCancel(url: string) {
  const controller = new AbortController();

  // Cancel after 5 seconds
  const timeoutId = setTimeout(() => controller.abort(), 5000);

  try {
    const res = await fetch(url, {
      signal: controller.signal   // Pass signal to fetch
    });
    clearTimeout(timeoutId);
    return await res.json();
  } catch (e) {
    if (e.name === 'AbortError') {
      console.log('Request was cancelled!');
    }
    throw e;
  }
}
Understanding Promise Microtask Queue
console.log('1 - synchronous');

setTimeout(() => console.log('4 - macrotask'), 0);

Promise.resolve().then(() => console.log('3 - microtask'));

console.log('2 - synchronous');

// Output ORDER:
// 1 - synchronous   (call stack, runs first)
// 2 - synchronous   (call stack, runs second)
// 3 - microtask     (Promise callbacks run BEFORE macrotasks)
// 4 - macrotask     (setTimeout callbacks run AFTER microtasks)
🧠
Promises use the microtask queue, which is always drained before the macrotask queue (setTimeout, setInterval). That's why Promise callbacks always run before setTimeout(fn, 0).
Mutex — Prevent Race Conditions
// When multiple async operations compete for shared resource
class Mutex {
  private queue: Array<() => void> = [];
  private locked = false;

  async acquire(): Promise<() => void> {
    if (this.locked) {
      await new Promise<void>(r => this.queue.push(r));
    }
    this.locked = true;
    return () => {
      const next = this.queue.shift();
      if (next) next(); else this.locked = false;
    };
  }
}

// Usage — serialize token refresh calls
const mutex = new Mutex();
async function refreshToken() {
  const release = await mutex.acquire();
  try {
    await doTokenRefresh();
  } finally {
    release();
  }
}

The 8 Commandments of Async
1
async functions always return a Promise
Even if you return a plain value like "hello", it gets auto-wrapped: Promise.resolve("hello").
2
await can ONLY be used inside an async function
Using await outside will cause a SyntaxError. The function containing await must be marked async.
3
await pauses ONLY the current async function
It does NOT block the entire JavaScript engine. Other code can still run while you're awaiting.
4
Missing await = you get a Promise, not the value
If you log without await: Promise { <pending> }. Always await before using the value.
5
forEach does NOT work with async/await
Use for...of (sequential) or Promise.all(array.map(...)) (parallel) instead.
6
Use Promise.all for independent parallel operations
If operations don't depend on each other, run them in parallel for 3x+ performance gains.
7
Always handle errors — async errors are silent by default
Unhandled Promise rejections can silently kill your tests. Use try/catch or .catch().
8
In Playwright, EVERY browser action is async — always await
page.click(), page.goto(), page.fill(), expect().toBeVisible() — all need await.

TypeScript Deep-Dive
Do I have to write Promise<string>?

Short answer: No — TypeScript infers it automatically. But there are clear rules for WHEN to write it explicitly, and why it matters for teams and libraries.

The Core Rule — Inference vs Explicit

TypeScript is smart. When you write async function and return a value, TypeScript automatically infers the return type. You do NOT need to write it — but you can to be explicit.

Without annotation (inferred)
// TypeScript figures it out itself
async function getGreeting() {
  return "hello";
}
// TS automatically knows this is:
// Promise<string>
// You don't write it — TS reads your return value
With annotation (explicit)
// You TELL TypeScript what to expect
async function getGreeting(): Promise<string> {
  return "hello";
}
// Both are identical at runtime.
// Explicit = documentation + safety net
✅
Both versions compile to identical JavaScript. The annotation only exists in TypeScript — it vanishes after compilation. It's purely for developer safety and readability.
How TypeScript Infers the Return Type

TypeScript looks at what you return inside the function and wraps it in Promise<...> automatically:

// You return string → TS infers Promise<string>
async function getName() {
  return "Sumit";                   // → Promise<string>
}

// You return number → TS infers Promise<number>
async function getAge() {
  return 28;                          // → Promise<number>
}

// You return nothing → TS infers Promise<void>
async function saveData() {
  await writeToFile();
  // no return  → Promise<void>
}

// You return an object → TS infers the shape
async function getUser() {
  return { name: "Sumit", role: "QA" };  // → Promise<{name: string, role: string}>
}

// You return string OR null → TS infers Promise<string | null>
async function findUser(id: number) {
  if (id < 0) return null;
  return "Sumit";                     // → Promise<string | null>
}
🧠
TypeScript inference is smart but not magic. It reads every return statement in your function and builds a union type of all possible return values, then wraps it in Promise<...>.
The 4 Scenarios — When to Write It, When to Skip

Scenario 1 — Simple private function → SKIP annotation

// Small helper inside your test file → inference is fine
async function waitASecond() {
  await new Promise(r => setTimeout(r, 1000));
}
// TS infers Promise<void> → no need to write it

Scenario 2 — Exported/public function → WRITE annotation

// This is in a shared helper file used by your whole team
// Being explicit = self-documenting, catches bugs early
export async function fetchOrderById(id: number): Promise<Order> {
  const res = await fetch(`/api/orders/${id}`);
  return res.json();
}
// ✅ If you accidentally return the wrong type,
//    TypeScript will error RIGHT HERE, not at the caller

Scenario 3 — Complex branching → WRITE annotation as safety net

// Multiple return paths — annotation acts as a contract
async function processOrder(id: number): Promise<string> {
  if (id < 0) return "invalid";

  const order = await fetchOrder(id);
  if (!order) return "not found";

  return order.status;
}
// If you accidentally return a number somewhere,
// TS catches it immediately: "number is not assignable to string"

Scenario 4 — Interface / implements → MUST write annotation

// When your class implements an interface, you MUST match the signature
interface UserService {
  getUser(id: number): Promise<User>;
}

class ApiUserService implements UserService {
  // MUST write Promise<User> to satisfy the interface
  async getUser(id: number): Promise<User> {
    const res = await fetch(`/users/${id}`);
    return res.json();
  }
}
All Possible Return Type Annotations

Here's every type you'll see on async functions:

// Returns nothing

  await someTask();
}

// Returns a primitive
async function getCount(): Promise<number> { return 42; }

// Returns a custom type / interface

  return { id: 1, status: "shipped" };
}

  return fetchOrder(id);

  const res = await fetch("/api/orders");

The "Safety Net" Effect — Why Explicit Wins for Shared Code

This is the most important reason to write the annotation explicitly. Watch what happens:

// ❌ WITHOUT explicit return type
async function getStatus(id: number) {
  if (id === 0) return 404;       // 🐛 oops, returned number
  return "active";               // intended string
}
// TS infers: Promise<string | number> — no error here!
// The bug travels silently to whoever CALLS this function

// ✅ WITH explicit return type
async function getStatus(id: number): Promise<string> {
  if (id === 0) return 404;       // 🔴 TS ERROR immediately!
  // "Type 'number' is not assignable to type 'string'"
  return "active";
}
// Bug is caught RIGHT HERE where it was introduced ✅
🔥
Team rule of thumb: Any function that is exported, part of a Page Object, in a helper/utility file, or used by more than one test — write the return type explicitly. It's free documentation and a compiler-enforced contract.

WebSocket in Plain JavaScript — The Basics

You'll rarely write raw WebSocket code in Playwright, but understanding it explains WHY Playwright works the way it does:

// Opening a WebSocket connection
const socket = new WebSocket('ws://localhost:9222/devtools/page/abc');

// 4 events — that's all there is to WebSocket!
socket.onopen = () => {
  console.log('✅ Connected! Handshake done once.');

  // Send a CDP command to click an element
  socket.send(JSON.stringify({
    id: 1,
    method: 'Input.dispatchMouseEvent',
    params: { type: 'mousePressed', x: 100, y: 200 }
  }));
};

socket.onmessage = (event) => {
  // Browser replied! Both sides can send anytime
  const data = JSON.parse(event.data);
  console.log('Browser says:', data);
};

socket.onerror = (err) => console.error('WS Error:', err);
socket.onclose = () => console.log('Connection closed');

// Close when done (Playwright does this after test suite)
socket.close();
💡
Playwright wraps ALL of this internally. When you write await page.click('#btn'), Playwright sends a WebSocket message to the browser and awaits the reply — you just see clean async/await.

Bonus: Testing WebSocket Apps WITH Playwright

Playwright can also test your own WebSocket-based application — intercept, mock, and assert on WS messages:

import { test, expect } from '@playwright/test';

test('websocket messages are received', async ({ page }) => {

  // Collect all WS messages from your app
  const messages: string[] = [];

  page.on('websocket', (ws) => {
    console.log('WS opened:', ws.url());

    ws.on('framesent',     (f) => console.log('→ Sent:', f.payload));
    ws.on('framereceived',  (f) => {
      console.log('← Recv:', f.payload);
      messages.push(f.payload as string);
    });
    ws.on('close',           () => console.log('WS closed'));
  });

  await page.goto('https://my-chat-app.com');
  await page.fill('#msg-input', 'Hello!');
  await page.click('#send-btn');

  // Wait until a specific WS message is received
  await page.waitForEvent('websocket');
  expect(messages.some(m => m.includes('Hello!'))).toBe(true);
});

⏱️ Most Common Junior Mistake
Timeouts & Waiting — Do It Right

The #1 source of flaky tests. Juniors reach for waitForTimeout(3000) like it's a solution. It's a band-aid. Learn the RIGHT waiting strategy for every scenario.

The Anti-Pattern Everyone Uses First
❌ Hard sleep — never do this
await page.click('#submit');
// "I'll just wait 3 seconds, should be fine"
await page.waitForTimeout(3000);
await page.click('#next-step');

// Problems:
// - Fast CI: 3s wasted when ready in 200ms
// - Slow CI: still fails if takes 4s
// - 10 tests × 3s = 30 extra seconds!
✅ Smart wait — wait for actual condition
await page.click('#submit');
// Wait for the RESULT of the click
await page.waitForSelector('#next-step',
  { state: 'visible' }
);
await page.click('#next-step');

// Proceeds the INSTANT it's ready
// Robust on fast AND slow machines
⚠️
waitForTimeout is only acceptable for: debugging locally, or intentional delays like "wait for animation to finish". Never use it as a proxy for "wait until page is ready".

waitForResponse — Intercept & Assert API calls

This is the most powerful wait pattern for QA engineers — you can wait for AND assert on the actual API call triggered by a UI action:

test('order placed triggers correct API call', async ({ page }) => {

  await page.goto('/checkout');

  // ⚠️ CRITICAL: set up the listener BEFORE the action that triggers it
  const responsePromise = page.waitForResponse(
    (res) => res.url().includes('/api/orders') && res.status() === 201
  );

  // THEN trigger the action
  await page.click('#place-order');

  // THEN await the response — resolves when the API call completes
  const response = await responsePromise;
  const body     = await response.json();

  expect(response.status()).toBe(201);
  expect(body.orderId).toBeDefined();
});
🚨
Always register waitForResponse / waitForRequest before the action that triggers them. If you register after, the response might already have fired and you'll wait forever.
expect() has built-in auto-waiting — use it!

Playwright's expect() assertions automatically retry until the condition is met (default 5s). You often don't need an explicit wait before asserting:

// ❌ Unnecessary double wait
await page.waitForSelector('.success-msg');
await expect(page.locator('.success-msg')).toBeVisible();

// ✅ expect() retries automatically — one line
await expect(page.locator('.success-msg')).toBeVisible();

// ✅ Custom timeout per assertion if needed
await expect(page.locator('.slow-widget'))
  .toBeVisible({ timeout: 15000 });

// ✅ Global timeout in playwright.config.ts
// expect: { timeout: 10000 }  ← sets default for all expects

The 3-Queue Mental Model
📚
Call Stack
Currently running code. LIFO — last in, first out. Only one frame runs at a time.
⚡
Microtask Queue
Promise callbacks (.then, .catch, await). Runs completely before any macrotask. Highest priority.
⏰
Macrotask Queue
setTimeout, setInterval, DOM events, I/O. Runs one task per Event Loop tick. Lower priority.
🧠
The golden rule: After every macrotask, the Event Loop drains the entire microtask queue before picking the next macrotask. That's why Promise.resolve().then() always fires before setTimeout(fn, 0).
await under the hood — it's just .then()
// These two are IDENTICAL under the hood:

// async/await style
async function withAwait() {
  const data = await fetchUser();
  console.log(data);
}

// What JS actually compiles it to (Promise .then)
function withThen() {
  return fetchUser().then((data) => {
    console.log(data);
  });
}

// When you write 'await', JS:
// 1. Suspends the current async function
// 2. Puts the rest of the function as a microtask callback
// 3. Returns control to the Event Loop
// 4. When the Promise resolves → microtask queue → resumes function

Full Chain: Auth Token → API → DB Verify

This is the exact pattern for X4V Unite style testing — get a token, call the pricing API, verify the result in the DB:

import { test, expect } from '@playwright/test';
import { Client } from 'pg';  // PostgreSQL client

test('pricing rule applies and is persisted in DB', async ({ request }) => {

  // ─── STEP 1: Get auth token ───────────────────────────────
  const authRes = await request.post('/auth/token', {
    data: { username: 'qauser', password: 'secret' }
  });
  expect(authRes.status()).toBe(200);
  const { access_token } = await authRes.json();

  // ─── STEP 2: Call pricing API with token ──────────────────
  const priceRes = await request.post('/api/pricing/calculate', {
    headers: { 'Authorization': `Bearer ${access_token}` },
    data: { productId: 'X4V-001', quantity: 10, ruleId: 'R-07' }
  });
  expect(priceRes.status()).toBe(200);
  const { finalPrice, transactionId } = await priceRes.json();
  expect(finalPrice).toBeGreaterThan(0);

  // ─── STEP 3: Verify in PostgreSQL DB ─────────────────────
  const db = new Client({ connectionString: process.env.DB_URL });
  await db.connect();

  try {
    const result = await db.query(
      `SELECT final_price, rule_id, status
         FROM pricing_transactions
        WHERE transaction_id = $1`,
      [transactionId]
    );

    expect(result.rows.length).toBe(1);
    expect(result.rows[0].final_price).toBe(finalPrice);
    expect(result.rows[0].status).toBe('COMMITTED');
  } finally {
    await db.end();  // always close DB connection!
  }
});
✅
The finally block closing the DB connection is critical. If your assertion throws, without finally the DB connection leaks — over many tests this can exhaust the connection pool.
Reusable API Helper Class — Your Team's Shared Utility
// api-helper.ts — wrap all API calls with token management
export class ApiHelper {
  private token: string | null = null;

  constructor(
    private request: APIRequestContext,
    private baseUrl: string
  ) {}

  // Lazy token — fetches once and reuses
  private async getToken(): Promise<string> {
    if (this.token) return this.token;
    const res  = await this.request.post(`${this.baseUrl}/auth/token`, {
      data: { username: process.env.API_USER, password: process.env.API_PASS }
    });
    const data = await res.json();
    this.token = data.access_token;
    return this.token!;
  }

  async post<T>(path: string, body: unknown): Promise<{ status: number; data: T }> {
    const token = await this.getToken();
    const res   = await this.request.post(`${this.baseUrl}${path}`, {
      headers: { 'Authorization': `Bearer ${token}` },
      data: body
    });
    return { status: res.status(), data: await res.json() as T };
  }

  async get<T>(path: string): Promise<{ status: number; data: T }> {
    const token = await this.getToken();
    const res   = await this.request.get(`${this.baseUrl}${path}`, {
      headers: { 'Authorization': `Bearer ${token}` }
    });
    return { status: res.status(), data: await res.json() as T };
  }
}

// Usage in test
test('pricing rule R-07', async ({ request }) => {
  const api = new ApiHelper(request, 'https://api.x4v.com');
  const { status, data } = await api.post<PriceResult>('/pricing/calculate', {
    productId: 'X4V-001', ruleId: 'R-07'
  });
  expect(status).toBe(200);
  expect(data.finalPrice).toBeGreaterThan(0);
});

🧪 When It Goes Wrong
Debugging Async Code Like a Pro

Async bugs are sneaky. A test passes locally, fails in CI. An await is missing and the error points to the wrong line. These techniques will save you hours.

Reading Async Stack Traces

Normal stack traces show where the error happened. Async stack traces show the chain of awaits that led there. Here's how to read them:

TimeoutError: page.click: Timeout 30000ms exceeded.
at LoginPage.clickSubmit (pages/LoginPage.ts:24:5) ← where it threw
at async LoginPage.login (pages/LoginPage.ts:18:5) ← async caller
at async Object.<anonymous> (tests/login.spec.ts:12:3) ← test that called it
1
Read bottom-up
The bottom line is your test. The top line is where the error actually occurred. Follow the chain upward.
2
Look for "async" in the trace
Lines marked async are await suspension points. The error crossed async boundaries to get to you.
3
The first non-node_modules line is yours
Skip Playwright internals. Find your file path — that's the code you need to fix.

Console Tricks for Async Debugging
// 1. console.time — measure how long an await takes
console.time('api-call');
const data = await fetchOrders();
console.timeEnd('api-call');  // api-call: 234.123ms

// 2. console.trace — see the async call chain
async function debugMe() {
  console.trace('Who called me?');
  await doWork();
}

// 3. Inspect a Promise without awaiting it
const promise = fetchUser(1);
console.log(promise);  // Promise { <pending> }  ← not awaited!
console.log(await promise); // { id: 1, name: "Sumit" } ← awaited!

// 4. Log every step of a chain to find where it breaks
const token = await getToken();
console.log('✅ Got token:', token.substring(0, 20));

const res = await callApi(token);
console.log('✅ API status:', res.status);

const dbRow = await verifyDb(res.id);
console.log('✅ DB row:', dbRow);
// When it fails, the last ✅ tells you exactly where the chain broke
Playwright-Specific Debug Tricks
// 1. PWDEBUG=1 — opens Playwright Inspector (step through actions)
//    Terminal: PWDEBUG=1 npx playwright test login.spec.ts

// 2. page.pause() — pause test AT a specific await point
await page.goto('/login');
await page.pause();  // ← test freezes here, Inspector opens
await page.click('#submit');

// 3. Screenshot on failure — put in catch
try {
  await page.click('#flaky-btn');
} catch (e) {
  await page.screenshot({
    path: `debug-${Date.now()}.png`,
    fullPage: true
  });
  throw e;
}

// 4. Slow down all actions for visual debugging
// playwright.config.ts:
// use: { launchOptions: { slowMo: 500 } }  ← 500ms between each action

// 5. Log all console errors from the browser
page.on('console', (msg) => {
  if (msg.type() === 'error')
    console.log('Browser error:', msg.text());
});
The Async Debug Checklist
1
Getting Promise { <pending> }?
You forgot await. Add it before the function call.
2
Test passes locally, fails in CI?
Timing issue. Replace waitForTimeout with waitForSelector or waitForResponse.
3
Error points to wrong line?
Read the async stack trace bottom-up. The real culprit is higher in the chain.
4
forEach loop not waiting?
Switch to for...of with await, or Promise.all(array.map(...)).
5
Multiple elements matched?
Use .first(), .nth(n), or a more specific selector.
6
Intermittent TimeoutError?
Add page.pause() just before the failing line. Inspect what the page looks like at that moment.

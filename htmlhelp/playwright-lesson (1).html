⚖️Playwright vs Selenium Philosophy ▶
🎭 Playwright
  • Auto-waits on every action
  • Built-in API testing
  • TypeScript / JS native
  • Browser contexts (isolated sessions)
  • Network interception built-in
  • Trace Viewer for debugging
  • Single tool: UI + API + Report
🔵 Selenium + Java
  • Manual explicit/implicit waits
  • Needs RestAssured for API
  • Java, Python, many languages
  • Driver per instance
  • External libs for network
  • External reporting (Extent/Allure)
  • Multiple tools to stitch together
💡Your Selenium + TestNG + RestAssured maps 1:1. Playwright merges all three into one framework.
🔷TypeScript — 5 Concepts You Need TS Primer▶
ℹ️Don't learn all TypeScript upfront. Learn these 5 things — they cover 95% of what you'll write in Playwright tests.

1. Types

TypeScript
const username: string  = 'standard_user';
const retries:  number  = 3;
const headless: boolean = true;
Java
String  username = "standard_user";
int     retries  = 3;
boolean headless = true;

2. Interface (like a Java POJO / data class)

interface User {
  username: string;
  password: string;
  role?:    string;   // optional field — no @Nullable needed
}
const u: User = { username: 'standard_user', password: 'secret_sauce' };

3. async / await — replaces WebDriverWait entirely

TypeScript / Playwright
test('login', async ({ page }) => {
  await page.goto('https://www.saucedemo.com');
  await page.fill('#user-name', 'standard_user');
  await page.fill('#password',  'secret_sauce');
  await page.click('#login-button');
});
Java / Selenium
@Test
public void login() {
  driver.get("https://www.saucedemo.com");
  driver.findElement(By.id("user-name"))
        .sendKeys("standard_user");
  driver.findElement(By.id("password"))
        .sendKeys("secret_sauce");
  driver.findElement(By.id("login-button")).click();
}

🛠️API Helper Utility — like Java BaseApiTest class Framework▶ 
// utils/apiHelper.ts
import { APIRequestContext, expect } from '@playwright/test';

export class ApiHelper {
  constructor(private request: APIRequestContext) {}

  async get<T>(url: string): Promise<T> {
    const res = await this.request.get(url);
    expect(res.ok()).toBeTruthy();
    return res.json();
  }

  async post<T>(url: string, data: object): Promise<{status: number; body: T}> {
    const res = await this.request.post(url, { data });
    return { status: res.status(), body: await res.json() };
  }
}

🎯All Locator Strategies — SauceDemo examples Core▶
💡Priority order: getByRole → getByLabel → getByText → getByPlaceholder → getByTestId → CSS → XPath

1. By Role (recommended)

page.getByRole('button', { name: 'Login' });
page.getByRole('heading', { name: 'Swag Labs' });
// Selenium: By.xpath("//button[normalize-space()='Login']")

2. By Text

page.getByText('Sauce Labs Backpack');
page.getByText(/sauce labs/i);   // regex, case-insensitive

3. By Label

page.getByLabel('Username');
page.getByLabel('Password');

4. By Placeholder

page.getByPlaceholder('Username');
// Selenium: By.cssSelector("input[placeholder='Username']")

5. By CSS Selector

page.locator('#user-name');
page.locator('.btn_action');
page.locator('input[type="text"]');
// Selenium: By.cssSelector("#user-name")

6. By XPath

page.locator('xpath=//div[@class="inventory_item_name"]');
// Selenium: By.xpath("//div[@class='inventory_item_name']")

7. By Test ID (data-testid)

page.getByTestId('login-button');
// HTML: <input data-testid="login-button">

8. Chaining + Filtering

// Click "Add to cart" inside a specific product card
page.locator('.inventory_item')
    .filter({ hasText: 'Sauce Labs Backpack' })
    .getByRole('button', { name: 'Add to cart' });

9. nth / first / last

page.locator('.inventory_item_name').first();
page.locator('.inventory_item_name').nth(2);  // 0-indexed
// Selenium: findElements(...).get(0)
⏳Waits & Auto-Waiting Intermediate▶
🚀Key advantage: Every click(), fill(), check() auto-waits for: visible → stable → enabled → not obscured. No more WebDriverWait + ExpectedConditions.
Playwright — zero wait code
await page.click('#login-button');
// Internally waits for all conditions
Selenium — manual wait
new WebDriverWait(driver, Duration.ofSeconds(10))
  .until(ExpectedConditions
    .elementToBeClickable(By.id("login-button")));
driver.findElement(By.id("login-button")).click();

When you do need explicit waits

await page.waitForURL('**/inventory.html');            // URL change
await page.waitForSelector('.inventory_list');          // element in DOM
await page.waitForResponse('**/api/products');          // network call
await page.locator('.loading').waitFor({ state: 'hidden' });  // spinner gone
await page.waitForFunction(() =>
  document.querySelectorAll('.inventory_item').length > 0
);

Assertions as smart waits (preferred)

// expect() retries for 5 seconds — use this over waitFor when possible
await expect(page.locator('.inventory_list')).toBeVisible();
await expect(page).toHaveURL(/inventory/);

📄Page Object Model — SauceDemo Intermediate▶

pages/LoginPage.ts

import { Page, Locator } from '@playwright/test';

export class LoginPage {
  private readonly usernameInput: Locator;
  private readonly passwordInput: Locator;
  private readonly loginBtn:      Locator;
  private readonly errorMsg:      Locator;

  constructor(private page: Page) {
    this.usernameInput = page.locator('#user-name');
    this.passwordInput = page.locator('#password');
    this.loginBtn      = page.locator('#login-button');
    this.errorMsg      = page.locator('[data-test="error"]');
  }

  async goto()   { await this.page.goto('/'); }

  async login(username: string, password: string) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.loginBtn.click();
  }

  async getErrorText(): Promise<string | null> {
    return this.errorMsg.textContent();
  }
}

import { Page, Locator, expect } from '@playwright/test';

export class InventoryPage {
  private readonly items: Locator;

  constructor(private page: Page) {
    this.items = page.locator('.inventory_item');
  }

  async assertLoaded() {
    await expect(this.page).toHaveURL(/inventory/);
    await expect(this.items).toHaveCount(6);
  }

  async addToCartByName(name: string) {
    await this.page
      .locator('.inventory_item')
      .filter({ hasText: name })
      .getByRole('button', { name: 'Add to cart' })
      .click();
  }
}

🔄The JavaScript Event Loop — Why Everything is Async Concept▶
ℹ️JavaScript runs on a single thread. It can't block and wait like Java's Thread.sleep(). Instead it uses an event loop to handle slow operations (network, timers, file I/O) without freezing.

Synchronous vs Asynchronous — core idea

JavaScript (single thread)
// JS cannot do this — would freeze the browser
const data = httpGet('https://reqres.in/api/users');
// Instead JS says: "start the request, tell me when done"
// → uses callbacks, Promises, or async/await
Java (multi-thread)
// Java can block a thread — others keep running
Response r = given().get("/api/users");
// Thread BLOCKS here until response arrives
System.out.println(r.getBody());

Evolution: Callback → Promise → async/await

// ── STAGE 1: Callbacks (old way — "callback hell") ──
fetchUser(1, (err, user) => {
  fetchOrders(user.id, (err, orders) => {
    fetchInvoice(orders[0].id, (err, invoice) => {
      console.log(invoice); // 3 levels deep — hard to read
    });
  });
});
// ── STAGE 2: Promises (cleaner chaining) ──
fetchUser(1)
  .then(user    => fetchOrders(user.id))
  .then(orders  => fetchInvoice(orders[0].id))
  .then(invoice => console.log(invoice))
  .catch(err   => console.error(err));
// ── STAGE 3: async/await (reads like Java sync code) ──
async function run() {
  const user    = await fetchUser(1);
  const orders  = await fetchOrders(user.id);
  const invoice = await fetchInvoice(orders[0].id);
  console.log(invoice);
}
// async/await is just syntax sugar over Promises
📦Promises — What They Are + All States Core▶
💡A Promise is an object representing a value that isn't available yet but will be resolved (success) or rejected (failure) in the future. Think of it like a Future<T> in Java.

Promise states

// A Promise has 3 states:
// ⏳ pending   — operation in progress
// ✅ fulfilled — operation succeeded → .then() fires
// ❌ rejected  — operation failed    → .catch() fires

const p = new Promise<string>((resolve, reject) => {
  const success = true;
  if (success) {
    resolve('Data loaded!');   // → fulfilled
  } else {
    reject(new Error('Failed!')); // → rejected
  }
});

p.then(data  => console.log(data))   // 'Data loaded!'
 .catch(err  => console.error(err))
 .finally(() => console.log('Done'));  // always runs

Creating a Promise-based delay (replaces Thread.sleep)

function delay(ms: number): Promise<void> {
  return new Promise(resolve => setTimeout(resolve, ms));
}

await delay(2000);  // pause 2 seconds (use sparingly in tests!)

Promise.all — run in parallel (like Java CompletableFuture.allOf)

// Run multiple API calls at the same time
const [user1, user2, user3] = await Promise.all([
  request.get('https://reqres.in/api/users/1'),
  request.get('https://reqres.in/api/users/2'),
  request.get('https://reqres.in/api/users/3'),
]);
// All 3 run simultaneously — faster than sequential await
// Java: CompletableFuture.allOf(f1, f2, f3).join()

Promise.allSettled — when you don't want one failure to stop others

const results = await Promise.allSettled([
  request.get('https://reqres.in/api/users/1'),
  request.get('https://reqres.in/api/users/999'), // 404
  request.get('https://reqres.in/api/users/3'),
]);
results.forEach(r => {
  if (r.status === 'fulfilled') console.log('OK:', r.value);
  else                           console.log('Failed:', r.reason);
});

async/await with try/catch — like Java try/catch

test('API with error handling', async ({ request }) => {
  try {
    const res  = await request.get('https://reqres.in/api/users/2');
    const body = await res.json();
    expect(body.data.id).toBe(2);
  } catch (err) {
    console.error('Request failed:', err);
    throw err; // fail the test
  } finally {
    console.log('Cleanup runs regardless'); // like @AfterMethod
  }
});

// utils/ApiClient.ts
import { APIRequestContext, expect } from '@playwright/test';

export class ApiClient {
  private baseUrl: string;

  constructor(private request: APIRequestContext, baseUrl: string) {
    this.baseUrl = baseUrl;
  }

  async get<T>(endpoint: string, params?: Record<string, any>): Promise<T> {
    const res = await this.request.get(`${this.baseUrl}${endpoint}`, { params });
    expect(res.ok(), `GET ${endpoint} failed with ${res.status()}`).toBeTruthy();
    return res.json();
  }

  async post<T>(endpoint: string, data: object): Promise<{status:number; body:T}> {
    const res = await this.request.post(`${this.baseUrl}${endpoint}`, { data });
    return { status: res.status(), body: await res.json() };
  }

  async delete(endpoint: string): Promise<number> {
    const res = await this.request.delete(`${this.baseUrl}${endpoint}`);
    return res.status();
  }
}

// Usage in test:

🗄️Sharding — What It Is + Playwright Test Sharding Advanced▶
ℹ️Sharding means splitting a large workload into smaller independent pieces (shards) that run simultaneously. It's used in both databases (horizontal scaling) and test suites (parallel CI execution).

Database Sharding — the general concept

// ── WHAT IS DATABASE SHARDING? ──
//
// A shard is a horizontal partition of data.
// Instead of one massive database, data is split
// across multiple database instances.
//
//  User IDs 1-1000   → DB Shard 1 (server A)
//  User IDs 1001-2000 → DB Shard 2 (server B)
//  User IDs 2001-3000 → DB Shard 3 (server C)
//
// Sharding key = the field used to decide which shard
// Common sharding keys: user_id, region, date range
//
// Benefits:
//   ✅ Horizontal scale — add more servers as data grows
//   ✅ Queries only touch relevant shard (faster)
//   ✅ Failure isolation — one shard down ≠ all down
//
// Challenges:
//   ❌ Cross-shard JOINs are complex
//   ❌ Re-sharding when shards fill up
//   ❌ Hotspots if shard key is poorly chosen

Sharding strategies

// 1. RANGE SHARDING — by value range
//    user_id 1-10000    → shard_01
//    user_id 10001-20000 → shard_02
//    Simple but can cause hotspots (new users all on latest shard)

// 2. HASH SHARDING — hash(user_id) % num_shards
//    hash(user_id=101) % 4 = 1  → shard_01
//    hash(user_id=202) % 4 = 2  → shard_02
//    Distributes evenly but range queries are hard

// 3. GEOGRAPHIC SHARDING — by region
//    India users     → shard_IN (Mumbai data center)
//    US users        → shard_US (Virginia data center)
//    EU users        → shard_EU (Frankfurt data center)
//    Good for data residency compliance (GDPR)

// 4. DIRECTORY SHARDING — lookup table maps key → shard
//    Flexible but lookup table becomes a bottleneck

Playwright Test Sharding — split test suite across machines

# .github/workflows/playwright-sharded.yml
jobs:
  playwright-shard:
    name: "Shard ${{ matrix.shardIndex }} / ${{ matrix.shardTotal }}"
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        shardIndex: [1, 2, 3, 4]
        shardTotal: [4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npx playwright install --with-deps chromium
      - run: npx playwright test
           --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: blob-report-${{ matrix.shardIndex }}
          path: blob-report/
          retention-days: 1   # short — merged next

  merge-reports:
    needs: playwright-shard
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - name: Download all blob reports
        uses: actions/download-artifact@v4
        with:
          path: all-blob-reports
          pattern: blob-report-*
          merge-multiple: true
      - name: Merge into single HTML report
        run: npx playwright merge-reports --reporter html ./all-blob-reports
      - uses: actions/upload-artifact@v4
        with:
          name: merged-playwright-report

🔐 Auth Setup — storageState (skip login in every test) Advanced ▶
🚀The most powerful config trick. Log in once, save browser state, reuse in every test. Massive speed boost — no login form on every test.
// auth.setup.ts — runs once before all tests
import { test as setup } from '@playwright/test';
import path from 'path';

const authFile = path.join(__dirname, 'playwright/.auth/user.json');

setup('authenticate', async ({ page }) => {
  await page.goto('https://www.saucedemo.com');
  await page.fill('#user-name', 'standard_user');
  await page.fill('#password',  'secret_sauce');
  await page.click('#login-button');
  await page.waitForURL('**/inventory.html');

  // Save cookies + localStorage to file
  await page.context().storageState({ path: authFile });
});

// Now all tests in 'chromium-auth' project start already logged in
// No need to call loginPage.login() in every @BeforeMethod

Why Playwright?

Playwright was built from scratch to support modern web apps — no Selenium inheritance, no hacks.

🔄
Auto-waiting
Automatically waits for elements to be actionable before performing any action. No more Thread.sleep().
🌐
Cross-Browser
Chromium, Firefox, and WebKit with a single API. One test, three engines.
📸
Visual Testing
Built-in screenshot and pixel-by-pixel visual comparison with expect(page).toHaveScreenshot().
🔗
API + UI Combined
Test REST APIs and UI in the same test. Set up state via API, verify via browser.
📦
Zero Config
Comes with built-in test runner, assertions, reporters, and browser downloads.
⚡
Parallel by Default
Tests run in parallel across workers. Isolated browser contexts for each test.

Setup & Config

Install Playwright, configure browsers, and wire up playwright.config.ts for multi-environment runs.

Installation

terminal BASH 
# Initialize new Playwright project
npm init playwright@latest

# Or add to existing project
npm install -D @playwright/test
npx playwright install        # Downloads browsers
npx playwright install-deps  # Linux system deps

playwright.config.ts

playwright.config.ts TS 
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: 'html',

  use: {
    baseURL: 'http://localhost:4200',  // Angular dev server
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },

  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit',   use: { ...devices['Desktop Safari'] } },
    { name: 'mobile-chrome', use: { ...devices['Pixel 5'] } },
  ],

  webServer: {  // Start Angular before tests
    command: 'ng serve',
    url: 'http://localhost:4200',
    reuseExistingServer: !process.env.CI,
  },
});
💡 Pro Tip: Use environment variables + .env files via dotenv for QA/UAT URL switching. Same pattern you already use in RestAssured configs!

Core Concepts

Architecture

Understand how Browser → BrowserContext → Page stack works — isolated, parallel, powerful.

Object Hierarchy

Browser
← Chromium / Firefox / WebKit process
BrowserContext
← Isolated session (cookies, storage). Like Incognito window.
Page
← Single browser tab. Has its own URL, DOM.
Locator
← Lazy element reference. Re-queried on every action.
📌 Key Insight: Each test automatically gets its own BrowserContext. This means tests are fully isolated — no shared cookies, localStorage, or state between tests. Perfect for parallel execution.

architecture-demo.spec.ts TS 
import { test, expect } from '@playwright/test';

// Each test() gets a fresh Page (and BrowserContext) automatically
test('example', async ({ page }) => {
  // page = isolated tab in a fresh context
  await page.goto('/');
});

// Creating contexts manually (for multi-user scenarios)
test('two users', async ({ browser }) => {
  const adminCtx = await browser.newContext({
    storageState: 'auth/admin.json'
  });
  const userCtx = await browser.newContext({
    storageState: 'auth/user.json'
  });

  const adminPage = await adminCtx.newPage();
  const userPage = await userCtx.newPage();

  // Both running in parallel, completely isolated
  await adminPage.goto('/admin/dashboard');
  await userPage.goto('/user/home');
});

Locators

Playwright's locators are lazy, auto-retrying, and strict by default. Forget stale element exceptions forever.

Preferred
CSS / XPath
Filtering

✅ Best Practice: Prefer these role-based & semantic locators. They match what users see, not implementation details.
preferred-locators.spec.tsTS
// By Role (best — uses ARIA roles)
page.getByRole('button', { name: 'Submit' })
page.getByRole('textbox', { name: 'Email' })
page.getByRole('link', { name: 'Sign in' })
page.getByRole('heading', { level: 1 })
page.getByRole('checkbox', { name: 'Remember me' })

// By Label (great for form fields)
page.getByLabel('Username')
page.getByLabel('Password')

// By Placeholder
page.getByPlaceholder('Enter your email')

// By Text
page.getByText('Welcome back')
page.getByText(/welcome/i)   // Regex

// By Test ID (explicit, for when nothing else works)
page.getByTestId('submit-btn')  // data-testid="submit-btn"

// By Alt Text (images)
page.getByAltText('Company logo')

// By Title
page.getByTitle('Close dialog')
⚠️ Use Sparingly: CSS/XPath work but are brittle. Prefer role-based locators above.
css-xpath.spec.tsTS
// CSS Selector
page.locator('button.primary')
page.locator('#login-form input[type="email"]')
page.locator('.pricing-table > .row:first-child')

// XPath
page.locator('xpath=//button[text()="Submit"]')
page.locator('//table//tr[contains(@class,"active")]')

// data-testid (via CSS)
page.locator('[data-testid="submit-btn"]')

// nth element
page.locator('li.item').nth(2)   // 0-indexed
page.locator('li.item').first()
page.locator('li.item').last()
filtering.spec.tsTS
// Filter by text content
page.getByRole('listitem')
    .filter({ hasText: 'Active' })

// Filter by child element
page.getByRole('listitem')
    .filter({ has: page.getByRole('button', { name: 'Edit' }) })

// Negate filter
page.getByRole('listitem')
    .filter({ hasNotText: 'Deleted' })

// Count matching elements
const rows = page.getByRole('row');
await expect(rows).toHaveCount(10);

// Loop through all matching
const items = await page.getByRole('listitem').all();
for (const item of items) {
  await expect(item).toBeVisible();
}
chaining.spec.tsTS
// Scope locator to a container (very powerful!)
const userCard = page.getByTestId('user-card');
await userCard.getByRole('button', { name: 'Edit' }).click();
await userCard.getByText('John Doe').isVisible();

// Locator inside locator
const table = page.locator('table.pricing');
const activeRow = table.locator('tr.active');
const price = activeRow.getByTestId('price');

// Frame handling
const frame = page.frameLocator('iframe#payment');
await frame.getByLabel('Card number').fill('4111111111111111');
Interactions

Actions & Events

Every action auto-waits for the element to be ready. Real browser events — not synthetic JS dispatches.

Common Actions

actions.spec.tsTS
// === NAVIGATION ===
await page.goto('https://example.com');
await page.goto('/login');               // Uses baseURL
await page.goBack();
await page.reload();
await page.waitForURL('**/dashboard');   // Wait for navigation

// === CLICK ===
await page.getByRole('button', { name: 'Login' }).click();
await page.getByText('Item').dblclick();
await page.getByText('Item').click({ button: 'right' });
await page.getByText('Item').click({ modifiers: ['Shift'] });

// === TYPE ===
await page.getByLabel('Email').fill('test@x4v.com');  // Fast fill
await page.getByLabel('Search').type('query');       // Simulates keypresses
await page.keyboard.press('Enter');
await page.keyboard.press('Tab');

// === CLEAR ===
await page.getByLabel('Username').clear();

// === SELECT ===
await page.getByLabel('Country').selectOption('India');
await page.getByLabel('Tags').selectOption(['A', 'B']);  // Multi-select

// === CHECKBOX / RADIO ===
await page.getByLabel('Remember me').check();
await page.getByLabel('Auto-renew').uncheck();
await page.getByLabel('Monthly').setChecked(true);

// === HOVER / FOCUS ===
await page.getByRole('button').hover();
await page.getByLabel('Email').focus();
await page.getByLabel('Email').blur();

// === FILE UPLOAD ===
await page.getByLabel('Upload').setInputFiles('path/to/file.pdf');

// === DRAG AND DROP ===
await page.getByTestId('source').dragTo(page.getByTestId('target'));

// === DIALOGS ===
page.on('dialog', dialog => dialog.accept());
await page.getByText('Delete').click();  // Triggers dialog → auto-accepted

expect() API

Assertions

Playwright assertions are async and auto-retrying. They poll until the condition is true or timeout expires.

Locator Assertions

assertions.spec.tsTS
const btn = page.getByRole('button', { name: 'Submit' });

// Visibility
await expect(btn).toBeVisible();
await expect(btn).toBeHidden();

// State
await expect(btn).toBeEnabled();
await expect(btn).toBeDisabled();
await expect(page.getByLabel('Remember me')).toBeChecked();

// Text Content
await expect(page.getByRole('heading')).toHaveText('Dashboard');
await expect(page.getByRole('heading')).toContainText('Dash');
await expect(page.getByRole('heading')).toHaveText(/Dashboard/i);

// Input value
await expect(page.getByLabel('Email')).toHaveValue('test@x4v.com');

// Attributes
await expect(btn).toHaveAttribute('type', 'submit');
await expect(btn).toHaveClass('primary-btn');
await expect(page.getByRole('listitem')).toHaveCount(5);

// CSS
await expect(btn).toHaveCSS('background-color', 'rgb(0, 123, 255)');

Page Assertions

page-assertions.spec.tsTS
// URL
await expect(page).toHaveURL('https://app.x4v.com/dashboard');
await expect(page).toHaveURL(/dashboard/);

// Title
await expect(page).toHaveTitle('X4V Unite Dashboard');

// Visual regression
await expect(page).toHaveScreenshot();
await expect(page.getByTestId('chart')).toHaveScreenshot();

// Soft assertions (don't stop on failure)
await expect.soft(page.getByText('Error')).toBeVisible();
await expect.soft(page.getByText('Retry')).toBeVisible();
// Will report ALL failures at end
🔄 Auto-retry: Assertions retry for up to expect.timeout (default 5s). This is different from Jest — you don't need waitFor wrappers!
Timing

Waits & Auto-wait

Playwright eliminates most explicit waits. But when you need them, here's the right way.

Golden Rule: If you find yourself writing page.waitForTimeout(3000), that's a smell. Use event-based waits instead.
waits.spec.tsTS
// ✅ Wait for navigation
await page.waitForURL('**/dashboard');
await page.waitForURL(/dashboard/);

// ✅ Wait for network
await page.waitForLoadState('networkidle');    // No requests for 500ms
await page.waitForLoadState('domcontentloaded');

// ✅ Wait for element
await page.getByRole('dialog').waitFor();
await page.getByRole('dialog').waitFor({ state: 'visible' });
await page.getByRole('spinner').waitFor({ state: 'hidden' });

// ✅ Wait for response (API call)
const responsePromise = page.waitForResponse('**/api/pricing/**');
await page.getByRole('button', { name: 'Calculate' }).click();
const response = await responsePromise;
expect(response.status()).toBe(200);

// ✅ Wait for request
const reqPromise = page.waitForRequest('**/api/rules');
await page.getByRole('button', { name: 'Load Rules' }).click();
await reqPromise;

// ✅ Wait for custom condition (function)
await page.waitForFunction(() =>
  document.querySelectorAll('.price-row').length > 5
);

// ❌ Avoid (hides real issues)
await page.waitForTimeout(3000);  // Never do this in real tests

API Testing

Full REST API testing without a browser. Combine with UI for hybrid flows — just like your RestAssured + Selenium stack.

Standalone API Tests

api.spec.tsTS
import { test, expect } from '@playwright/test';

test('GET pricing rules', async ({ request }) => {
  const res = await request.get('/api/v1/pricing/rules');

  expect(res.status()).toBe(200);
  expect(res.ok()).toBeTruthy();

  const body = await res.json();
  expect(body.rules).toHaveLength(30);    // X4V has 30 rules!
  expect(body.rules[0]).toHaveProperty('powerRank');
});

test('POST create rule', async ({ request }) => {
  const res = await request.post('/api/v1/pricing/rules', {
    data: {
      name: 'New Rule',
      powerRank: 15,
      conditions: [{ field: 'vehicleType', value: 'EV' }]
    },
    headers: { 'Authorization': `Bearer ${process.env.TOKEN}` }
  });
  expect(res.status()).toBe(201);

  const created = await res.json();
  expect(created.id).toBeDefined();
});

test('PUT / PATCH / DELETE', async ({ request }) => {
  await request.put('/api/v1/rules/1', { data: { powerRank: 5 } });
  await request.patch('/api/v1/rules/1', { data: { active: false } });
  const del = await request.delete('/api/v1/rules/1');
  expect(del.status()).toBe(204);
});

Hybrid: API Setup + UI Verify

hybrid.spec.tsTS
// Create data via API, verify via UI — MUCH faster than all-UI
test('created rule appears in UI', async ({ page, request }) => {
  // 1. Set up data via API (fast)
  const res = await request.post('/api/v1/pricing/rules', {
    data: { name: 'EV Discount Rule', powerRank: 10 }
  });
  const { id } = await res.json();

  // 2. Navigate to UI
  await page.goto(`/pricing/rules/${id}`);

  // 3. Assert UI shows the rule
  await expect(page.getByRole('heading')).toHaveText('EV Discount Rule');
  await expect(page.getByTestId('power-rank')).toHaveText('10');
});
💡 Auth via API: Log in via API request, save storageState, reuse in UI tests — skip the login UI entirely for 90% of your tests.

Fixtures & Hooks

Playwright fixtures are the evolution of TestNG @BeforeMethod. Typed, composable, and auto-scoped.

Built-in Fixtures

📄
page
Fresh browser page for each test. Auto-closes after test.
🌐
browser
Shared browser instance across tests in the same worker.
📦
context
Isolated browser context (fresh cookies/storage per test).
🔌
request
API request context. No browser needed.

Custom Fixtures

fixtures/index.tsTS
import { test as base, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';
import { PricingPage } from '../pages/PricingPage';

type Fixtures = {
  loginPage: LoginPage;
  pricingPage: PricingPage;
  authenticatedPage: Page;
};

export const test = base.extend<Fixtures>({
  // Auto-instantiate Page Objects
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },

  pricingPage: async ({ page }, use) => {
    await use(new PricingPage(page));
  },

  // Auto-login fixture
  authenticatedPage: async ({ page }, use) => {
    await page.goto('/login');
    await page.getByLabel('Email').fill(process.env.TEST_USER!);
    await page.getByLabel('Password').fill(process.env.TEST_PASS!);
    await page.getByRole('button', { name: 'Login' }).click();
    await page.waitForURL('**/dashboard');
    await use(page);  // ← Hand control to the test
    // Cleanup after test: nothing needed, context is auto-destroyed
  },
});

export { expect };

Design Pattern

Page Object Model

Same POM you know from Selenium, but with Playwright's lazy locators and TypeScript power.

Page Object Class

pages/LoginPage.tsTS
import { type Page, type Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;

  // Locators defined as class properties (lazy — not queried until used)
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitBtn: Locator;
  readonly errorMsg: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput    = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitBtn     = page.getByRole('button', { name: 'Sign In' });
    this.errorMsg      = page.getByRole('alert');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitBtn.click();
  }

  async loginAndWait(email: string, password: string) {
    await this.login(email, password);
    await this.page.waitForURL('**/dashboard');
  }
}
💡 Key difference from Selenium: Locators in Playwright are lazy — assigning this.submitBtn = page.getByRole(...) does NOT touch the DOM. The DOM query only fires when you call .click(), .fill(), etc. No stale element exceptions!

Migration Guide

Playwright vs Selenium

You know Selenium deeply. Here's exactly how concepts map — and where Playwright wins.

Concept Selenium + Java Playwright + TypeScript
Find element driver.findElement(By.id("x")) page.getByRole('button')
Click element.click() await locator.click()
Type text element.sendKeys("text") await locator.fill("text")
Explicit wait WebDriverWait(driver, 10) Not needed — auto-wait built in
Implicit wait driver.manage().timeouts() Auto-wait on every action
Select dropdown new Select(el).selectByText() await locator.selectOption()
Assert TestNG Assert.assertEquals() await expect(locator).toHaveText()
Before/After @BeforeMethod / @AfterMethod Fixtures (beforeEach implicit)
Parallel TestNG XML suites + threadLocal Workers auto-parallel (built-in)
Screenshot TakesScreenshot ts = ... await page.screenshot()
API testing RestAssured (separate library) Built-in via request fixture
Report Extent Reports / Allure Built-in HTML reporter

Key Advantages

🚫
No StaleElementException
Locators re-query the DOM on every action. Detached elements are automatically refetched.
🔐
No WebDriver Protocol
Communicates directly with browsers via CDP (Chrome) and remote protocols — faster and more reliable.
📦
Batteries Included
No separate RestAssured, Extent Reports, or parallel setup. One npm install.

TypeScript / JavaScript Concept

The ... Spread Operator

Used heavily in Playwright configs, fixtures, and POM. Understand it once and configs become obvious.

What does ... do?

The spread operator expands an object or array into individual pieces — like pouring a glass of water into a larger container.

📤
Object Spread
Copies all key-value pairs from one object into another. {...a, ...b} merges both — b wins on conflicts.
📋
Array Spread
Expands array items inline: [...arr1, ...arr2] concatenates. [...arr] shallow-clones.
📥
Rest (opposite)
Same syntax in destructuring means "collect the rest": const { a, ...rest } = obj — rest has everything except a.

Core Mechanics

spread-basics.tsTS
// ── Object spread ───────────────────────────────────────────────
const base = { timeout: 30000, headless: true };
const override = { headless: false, slowMo: 200 };

const merged = { ...base, ...override };
// → { timeout: 30000, headless: false, slowMo: 200 }
//   ↑ base props copied    ↑ override wins on 'headless'

// ── Shallow clone (no mutation) ──────────────────────────────────
const original = { user: 'sumit', role: 'qa' };
const clone = { ...original };        // new object, same values
clone.role = 'admin';               // original.role is still 'qa'

// ── Add/override single property ────────────────────────────────
const withExtra = { ...original, role: 'admin', verified: true };
// → { user: 'sumit', role: 'admin', verified: true }

// ── Array spread ─────────────────────────────────────────────────
const browsers = ['chromium', 'firefox'];
const all = [...browsers, 'webkit'];   // ['chromium', 'firefox', 'webkit']

// ── Rest (collecting remaining props) ───────────────────────────
const config = { baseURL: 'http://localhost', timeout: 5000, headless: true };
const { baseURL, ...rest } = config;
// baseURL = 'http://localhost'
// rest    = { timeout: 5000, headless: true }

In playwright.config.ts — devices spread

Context: devices['Desktop Chrome'] is a plain object with ~8 properties (viewport, userAgent, isMobile, etc.). The ... copies all of them into your project config, so you only override what you need.
playwright.config.tsTS
import { defineConfig, devices } from '@playwright/test';

// devices['Desktop Chrome'] expands to something like:
// {
//   viewport: { width: 1280, height: 720 },
//   userAgent: 'Mozilla/5.0 ...',
//   deviceScaleFactor: 1,
//   isMobile: false,
//   hasTouch: false,
// }

export default defineConfig({
  projects: [
    {
      name: 'chromium',
      use: {
        ...devices['Desktop Chrome'],  // ← spreads ALL device props
        baseURL: 'http://localhost:4200', // ← your override on top
      },
    },
    {
      name: 'mobile',
      use: {
        ...devices['Pixel 5'],           // viewport 393×851, touch: true
        baseURL: 'http://localhost:4200',
      },
    },
  ],
});

In Fixtures — spreading fixture sets

fixtures/combined.tsTS
// Compose multiple fixture sets — spread merges the type definitions
import { test as base } from '@playwright/test';
import { AuthFixtures, authFixtures } from './auth';
import { PageFixtures, pageFixtures } from './pages';

// Spread merges two separate fixture type definitions into one
export const test = base.extend<AuthFixtures & PageFixtures>({
  ...authFixtures,   // copies { authenticatedPage: ..., token: ... }
  ...pageFixtures,   // copies { loginPage: ..., dashboardPage: ... }
});

// Now tests can destructure from either set:
test('dashboard loads', async ({ authenticatedPage, dashboardPage }) => {
  // authenticatedPage from authFixtures
  // dashboardPage     from pageFixtures
});

Java vs TypeScript mental model

Java (what you know)TypeScript ... equivalent
new HashMap(existingMap){...existingObj} — shallow copy
map.putAll(otherMap){...map1, ...map2} — merge, last wins
new ArrayList(list)[...list] — array copy
Collections.addAll(a, b)[...a, ...b] — concatenate
No direct equivalentconst {x, ...rest} = obj — destructure + collect rest
💡 Key rule: Later properties win. { timeout: 5000, ...override } — override can change timeout. { ...override, timeout: 5000 } — your literal 5000 always wins regardless of override.
Fixture System Deep Dive

base.extend<T>

The engine behind all custom fixtures. Understand this pattern and you can build any test infrastructure Playwright is capable of.

The Signature Explained

anatomy.tsTS
// Full signature:
const test = base.extend<YourFixtureTypes>({
  fixtureName: async ({ /* other fixtures to inject */ }, use) => {
    // SETUP: runs before the test (like @BeforeMethod in TestNG)
    const value = await setupSomething();

    await use(value);  // ← HAND CONTROL to the test

    // TEARDOWN: runs after the test (like @AfterMethod)
    await cleanupSomething(value);
  },
});
🔤
<T> Generic Type
Tells TypeScript what custom fixtures you're adding. Autocomplete in tests works because of this. T is a plain object type with fixture names as keys.
📌
base Parameter
base is the imported test renamed. It carries all built-in fixtures. Your extension wraps it — you get page, request, etc. for free.
🔁
use(value) Pivot
Everything before await use() is setup. Everything after is teardown. Playwright guarantees teardown runs even if the test throws.
🔗
DI / Injection
Fixtures can depend on other fixtures. Playwright resolves the dependency graph automatically — just declare what you need in the destructured argument.

import { test as base } from '@playwright/test';

// Step 1: Declare the shape of YOUR fixtures as a TypeScript type
type MyFixtures = {
  baseURL: string;
  authToken: string;
};

// Step 2: extend base with that type
export const test = base.extend<MyFixtures>({

  // Simple string fixture — no setup/teardown needed
  baseURL: async ({}, use) => {
    await use(process.env.BASE_URL ?? 'http://localhost:4200');
  },

  // Token fetched once and passed in
  authToken: async ({ request }, use) => {
    const res = await request.post('/api/auth/login', {
      data: { email: process.env.TEST_USER, password: process.env.TEST_PASS }
    });
    const { token } = await res.json();
    await use(token);
    // No teardown needed — token just goes out of scope
  },
});

// Usage in test:
test('uses token', async ({ authToken, request }) => {
  await request.get('/api/protected', {
    headers: { Authorization: `Bearer ${authToken}` }
  });
});

fixtures/auth.tsTS
import { test as base, type Page } from '@playwright/test';

type AuthFixtures = { authenticatedPage: Page };

export const test = base.extend<AuthFixtures>({

  authenticatedPage: async ({ page }, use) => {
    // SETUP: log in before every test that requests this fixture
    await page.goto('/login');
    await page.getByLabel('Email').fill(process.env.TEST_USER!);
    await page.getByLabel('Password').fill(process.env.TEST_PASS!);
    await page.getByRole('button', { name: 'Sign In' }).click();
    await page.waitForURL('**/dashboard');

    await use(page);   // ← TEST RUNS HERE

    // TEARDOWN: optional - e.g. clear localStorage or save storage state
    await page.context().storageState({ path: 'auth/state.json' });
  },
});

// Pro tip: Skip the UI login entirely by reusing saved storageState:
export const fastTest = base.extend<AuthFixtures>({
  authenticatedPage: async ({ browser }, use) => {
    const ctx = await browser.newContext({
      storageState: 'auth/state.json'   // pre-saved cookies + localStorage
    });
    const page = await ctx.newPage();
    await use(page);
    await ctx.close();
  },
});
fixtures/worker-scope.tsTS
// Default scope: 'test' — re-created for every test
// Worker scope: created ONCE per worker process, shared across tests
// Use worker scope for expensive operations like: DB connections, auth token fetch

type WorkerFixtures = { dbConnection: DbClient };
// Worker fixtures are declared in the SECOND generic argument of extend<>

export const test = base.extend<{}, WorkerFixtures>({
  dbConnection: [async ({}, use) => {
    const client = await DbClient.connect({
      host: process.env.DB_HOST,
      database: 'x4v_test'
    });

    await use(client);   // All tests in this worker share this connection

    await client.disconnect();  // Runs once when worker shuts down
  }, { scope: 'worker' }],  // ← this options object sets the scope
});

// Analogy: like HikariCP pooling in your Java framework —
// one connection shared per thread instead of per test
fixtures/chained.tsTS
// Fixture A depends on Fixture B — Playwright resolves the chain
type ChainedFixtures = {
  authToken: string;
  apiClient: ApiClient;
  loggedInPage: Page;
};

export const test = base.extend<ChainedFixtures>({

  // Level 1: Fetch token
  authToken: async ({ request }, use) => {
    const res = await request.post('/api/auth/token', {
      data: { grant_type: 'client_credentials' }
    });
    await use((await res.json()).access_token);
  },

  // Level 2: Uses authToken fixture (injected automatically)
  apiClient: async ({ authToken, request }, use) => {
    const client = new ApiClient(request, authToken);
    await use(client);
  },

  // Level 3: Uses apiClient to seed data, then returns a page
  loggedInPage: async ({ apiClient, page }, use) => {
    await apiClient.seedTestData();    // set up via API
    await page.goto('/dashboard');
    await use(page);
    await apiClient.cleanupTestData(); // teardown
  },
});

// Playwright builds the dependency graph:
// request → authToken → apiClient → loggedInPage
// Each is set up in order, torn down in reverse
🔑 Mental model for Java devs: Think of base.extend<T> as building a typed DI container (like Spring context). The generic T is your bean definitions, each fixture implementation is a @Bean method, and await use(value) is the point where the test "runs" inside the container's lifecycle.

test() Function Anatomy

The test() function has three parts: title (string), optional details (annotations/tags), and body (async callback receiving fixtures + testInfo).
test-anatomy.spec.tsTS
import { test, expect } from '@playwright/test';

// Basic signature
test('title string', async ({ page, request }) => { /* body */ });

// With details (tags, annotations)
test('login flow',
  { tag: ['@smoke', '@regression'], annotation: { type: 'issue', description: 'X4V-123' } },
  async ({ page }, testInfo) => {
    testInfo.setTimeout(60000);  // override timeout for this test
    await page.goto('/');
  }
);

// Most common fixture args:
// { page }           → UI testing
// { request }        → API testing
// { page, request }  → Hybrid
// { browser }        → Manual context creation
// { context }        → Access current BrowserContext

Locator Objects — Key Takeaways

Locators are LAZY: page.getBy*() and page.locator() return Locator objects — they do NOT touch the DOM. Do not await the locator itself; only await the action (.click(), .fill()) or assertion on it.
lazy-locators.spec.tsTS
// ✅ Correct — locator is not awaited, only the action is
const btn = page.getByRole('button', { name: 'Submit' });  // no await
await btn.click();                                            // await here

// ❌ Wrong — locator is NOT a Promise, await does nothing useful
const btn2 = await page.getByRole('button');  // pointless await

AI Agents (Playwright Test Agents)

Three agents: Use them sequentially for best results. Planner → Generator → Healer is the recommended flow.
🗺️
Planner
In: seed test / PRD. Out: .md test plan file. Explores the application and generates a comprehensive test coverage plan.
⚙️
Generator
In: test plan .md. Out: *.spec.ts. Converts the plan into executable Playwright tests with boilerplate code.
🩹
Healer
In: failing spec file. Out: error-free spec. Fixes failing tests — can be used in isolation without running full pipeline.
⚠️ AI Agent Caveats (from course): Generated code needs manual review. Assertions may only check visibility, not business rules. Sensitive credentials/API keys must be reviewed. Framework compliance may need adjustment for your project structure.
